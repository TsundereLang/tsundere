Testing

Testing Tsundere projects means testing your `.yuri` source, generated runtime behavior, Discord handlers, services, installers, CLI commands, and docs links.

Overview

Tsundere supports normal JavaScript testing workflows because it compiles to Node-compatible output. The official testing package roadmap adds helpers for command contexts, interactions, components, modal submissions, cache data, and Discord REST mocks.

Why It Exists

Discord bots often fail in handlers that are hard to manually reproduce. Tests let you validate option parsing, permission paths, component data, warning databases, logging decisions, and edge cases before deploying a bot to real servers.

When To Use It

  • Use unit tests for pure services, config loading, and command helper functions.
  • Use integration tests for command handlers with mocked Discord contexts.
  • Use snapshot-like tests for generated command metadata.
  • Use CLI tests for templates, installer scripts, package commands, and runtime launches.

Unit Test Example

import { test, expect } from "@tsundere/testing"
import { formatDuration } from "../src/services/time"

test("formats minutes", () => {
  expect(formatDuration(5)).toBe("5 minutes")
})

The official API is planned. Today, you can use Node's test runner, Vitest, or Jest against generated JavaScript.

Command Handler Example

test("ping replies with latency", async () => {
  const ctx = mockCommandCtx({
    commandName: "ping",
    client: { ping: 42 }
  })

  await pingHandler(ctx)

  expect(ctx.replies[0].content).toContain("42ms")
})

Mock the smallest useful surface. A command test usually needs options, user, member, guild, reply calls, and permission state. It does not need a real gateway connection.

Command Metadata Test

import ban from "../src/commands/admin/ban"

test("ban command requires target", () => {
  const data = ban.toJSON()
  expect(data.name).toBe("ban")
  expect(data.options[0].name).toBe("target")
  expect(data.options[0].required).toBe(true)
})

Common Mistakes

  • Do not only test happy paths. Test missing options, missing permissions, deleted members, and partial cached data.
  • Do not call Discord's real API in normal unit tests.
  • Do not let tests depend on command propagation timing.
  • Do not snapshot huge generated files when targeted assertions would be clearer.

Performance Notes

Keep unit tests fast and run them on every commit. Put slow Discord-like integration tests behind a separate script or CI job. Cache compiler fixtures when testing many `.yuri` examples.
