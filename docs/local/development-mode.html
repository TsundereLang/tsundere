Getting Started

Development Mode

`tsundere dev` is the normal loop while building a bot. It discovers commands, compiles source, emits runtime JavaScript, launches Node, watches `.yuri` files, and restarts the runtime when code changes.

What Runs When You Type It

tsundere dev

The CLI loads `tsundere.config.json`, validates basic fields, and calls `writeCommandManifest`. That writes `.yuri-cache/discord.commands.json` based on the command discovery config. Then `devProject` builds the project and starts `.tsundere/runtime-build/main.js` with Node.

The watcher uses `fs.watch` when available and also keeps a polling snapshot every 750ms. That fallback matters on Windows and OneDrive-style folders where recursive watching can be unreliable.

Rebuilds are serialized. If a file changes while a build is already running, the dev loop marks a pending rebuild and runs it after the current build finishes. That avoids two compiler passes writing the same output folder at once. The restart delay is short, so saving several files quickly should settle into one or two restarts rather than a long chain of half-built runtime processes.

The Runtime Process

Runtime output is not run from `build/main.ts`. The dev flow writes JavaScript into `.tsundere/runtime-build`, then spawns Node with that entry. Standard output is forwarded directly. Standard error is passed through a small branding rewrite so Node version footers can say `Tsundere Runtime <version> (v...)`.

The `.env` file is read by `readDotEnv` in the compiler project module. Values from `.env` are merged into the spawned process environment, so `env.DISCORD_TOKEN` works after the compiler adds the `env` prelude.

When the child process exits because you changed files, dev mode kills it with a termination signal and immediately rebuilds. When it exits by itself with a non-zero code, dev mode prints `Tsundere runtime exited with code ... Waiting for changes...` and keeps the watcher alive. That is deliberate: you can fix the source file, save, and let the same dev session recover.

Command Discovery During Dev

The default starter config points discovery at `./src/commands`, uses route-based names, and includes an admin group override. A file like `src/commands/ping.yuri` becomes `/ping`; a file under an admin group can become `/admin ban` depending on the group settings.

tsundere commands sync

Use the command above when you want to regenerate command metadata without starting dev mode.

Discovery does not execute command files. It reads `.yuri` source and extracts simple metadata from object-style `name` and `description` fields or from `Slash.command("...")` and `.description("...")` builder calls. If you compute a command name dynamically, discovery will not understand it. Keep exported command names literal if you want clean metadata.

Expected Files After Dev Starts

After the first successful dev build, you should be able to inspect `build`, `.tsundere/runtime-build`, `.tsundere/runtime/discord`, and `.yuri-cache/discord.commands.json`. The exact files depend on your `source` tree, but a Discord starter should at least produce a runtime `main.js`. If those folders are missing, the build step did not complete.

On Windows and Linux the paths are the same inside the project. The difference is only in how the runtime process and platform checks are launched. Windows npm commands are wrapped as `.cmd` executables where needed. Linux uses shell-compatible command lookup and may apply executable permissions to scripts during doctor checks.

Troubleshooting Dev Mode

  • If dev mode says no `.yuri` files were found, check the `source` field in `tsundere.config.json`.
  • If the runtime exits immediately, run `tsundere build` to separate compile errors from runtime errors.
  • If Discord events do not fire, check intents in both code and the Discord Developer Portal.
  • If command metadata is stale, run `tsundere commands sync` and inspect `.yuri-cache/discord.commands.json`.
