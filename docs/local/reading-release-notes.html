Getting Started

Reading Release Notes

Release notes matter because Tsundere changes quickly. Updates can affect the compiler, the bundled Discord runtime, installer assets, the VS Code extension, package optimizer behavior, and the updater itself.

Where Notes Live

The repo includes `updates.md` for Discord-friendly project updates and `docs/local/updates.html` for local docs readers. GitHub releases are the source used by the updater command. The updater reads the latest release tag, release page URL, assets, and body text from GitHub.

tsundere updater info

The release notes you read should match the install path you use. `updates.md` can explain ongoing project work, but the updater can only install what is attached to a GitHub release. If a note says a CLI release is available, verify that the release has an installable tarball asset before telling users to run the updater.

What To Look For

  • CLI tarball asset names such as `tsundere-cli-0.1.1.tgz` or `tsundere-cli.tgz`.
  • Compiler changes that may affect `.yuri` syntax lowering.
  • Discord runtime changes, especially around gateway login, interactions, components, role helpers, and command sync.
  • Installer changes for Windows, Linux, WSL, VS Code, Cursor, or Antigravity.
  • Security update language. The CLI can show a security update notice when release text includes security-related keywords.

Also look for migration notes. A change to `tsundere.config.json` defaults, `.yuri` lowering, package optimizer lock snapshots, or the local Discord runtime can require project cleanup even when the CLI update itself succeeds. Good release notes should tell you whether to rerun `tsundere install`, `tsundere runtime install`, `tsundere commands sync`, or a clean build.

Before Updating a Bot

For serious bots, read the release notes before running `tsundere updater self --yes`. Then update in a test copy of the project, run install, build, command sync, and dev/start smoke checks.

tsundere install
tsundere build
tsundere commands sync
tsundere doctor

If the release touches Discord runtime behavior, also run the bot in `tsundere dev` and test at least one slash command, one interaction reply, and any role or welcome flows your bot depends on. If the release touches package management, compare npm's `package-lock.json` with Tsundere's YAML snapshots after install.

How The Updater Chooses Assets

The updater prefers a versioned tarball such as `tsundere-cli-0.1.1.tgz`, then falls back to `tsundere-cli.tgz`, then any asset matching `tsundere-cli*.tgz`. That means release asset names are not cosmetic. A release can have beautiful notes and still be impossible for `tsundere updater self` to install if the asset name is wrong.

Version comparison is simple numeric dot comparison after trimming a leading `v`. Pre-release naming should be tested carefully because the updater is optimized for stable tags such as `v0.1.1`. Use `tsundere updater self --dry-run` to prove selection before recommending an update publicly.

Incomplete or Broken Releases

If a release is missing `tsundere-cli.tgz` or a versioned CLI tarball, `tsundere updater self` reports that the latest release is missing a CLI tarball asset. In that case, wait for a corrected release or install from a verified source checkout.

Do not fix a broken release by changing project code. Fix the release layout, attach the expected artifacts, and publish a corrected release. Project-level commands cannot create missing GitHub release assets.
