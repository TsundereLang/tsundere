Getting Started

Updating Tsundere

Tsundere has two different update paths: updating the CLI itself through GitHub releases, and updating packages inside a project through npm-compatible commands.

Check the Installed CLI

tsundere version
tsundere updater
tsundere updater info

`tsundere version` reads the installed package version and prints the configured GitHub release repo. `tsundere updater` checks `https://api.github.com/repos/TsundereLang/tsundere/releases/latest` by default. You can override that repo with `TSUNDERE_UPDATE_REPO`.

The updater compares release tags after removing a leading `v`, so `v0.1.1` and `0.1.1` compare as the same version. If GitHub cannot be reached, the updater returns a failure message rather than guessing. Network access, GitHub rate limits, or a wrong `TSUNDERE_UPDATE_REPO` value can all produce the same practical result: no release metadata.

Self Update

tsundere updater self --dry-run
tsundere updater self --yes

Self update looks for a release asset matching `tsundere-cli-<version>.tgz`, then `tsundere-cli.tgz`, then any `tsundere-cli*.tgz`. Without `--yes`, it reports the update but does not install. With `--dry-run`, it reports what would happen without changing the machine.

The installer uses npm global install for the downloaded tarball. If global npm permissions are broken, the update will fail even if the release was found.

Use `--dry-run` before trusting a new release layout. It proves the updater can find the release and select the tarball asset without changing the installed CLI. Use `--yes` only after the selected asset looks right. The downloaded tarball is written to a temporary update folder and removed after the install attempt.

Daily Update Checks

tsundere updater cron install
tsundere updater cron status
tsundere updater cron remove

On Windows, this creates or removes a Scheduled Task named `Tsundere Daily Update Check`. On Unix-like systems, it edits the current crontab with a marked `tsundere updater check` line.

The automated check is a notification/check mechanism. It does not silently install a new CLI every day. The scheduled command runs `updater check`, so you still choose when to run `tsundere updater self --yes`. This is safer for projects because a CLI update can change build behavior, runtime output, or installer layout.

On Windows, scheduled tasks use `schtasks`. On Unix-like systems, the updater rewrites the user's crontab after removing any previous line with the `# tsundere-daily-update-check` marker. If your crontab already has important entries, the updater keeps them and only manages its marked line.

Updating Project Packages

tsundere update discord.js
tsundere add zod
tsundere remove zod

These commands go through `optimizedNpmInstall`, so npm still owns dependency resolution and `package-lock.json`. Tsundere adds store reuse, metrics, and YAML snapshots around npm.

Package updates are project-local. They do not update the Tsundere CLI. `tsundere update discord.js` maps to npm package update behavior inside the current project, while `tsundere updater self --yes` installs a new global Tsundere CLI tarball. Keep those two paths separate when debugging version mismatches.

Security Update Notices

Before commands such as `dev`, `start`, `build`, `install`, `add`, and `update`, the CLI can check whether the latest release looks like a security update. The result is cached for 24 hours in `~/.tsundere/security-update-cache.json`. Set `TSUNDERE_NO_UPDATE_NOTICE=1` when you need to suppress that notice in automation.

The notice is based on release text containing words such as security, critical, vulnerability, CVE, exploit, hotfix, or patch now. It is a helpful signal, not a formal security advisory system. Read the actual release notes before deciding how urgent an update is.
