Discord Commands

Commands are the primary interaction surface for modern Discord bots. Tsundere keeps command definitions close to source files and reduces deploy-script boilerplate without becoming a command framework.

Overview

A command file exports metadata that can be discovered, validated, synced, and handled by your bot. Tsundere supports slash command builders, options, subcommands, groups, permissions, guild-only commands, autocomplete metadata, and route-based discovery.

Why It Exists

Discord command setup usually repeats the same work: define commands, register them, map interactions to handlers, parse options, and keep stale commands cleaned up. Tsundere turns those steps into an understandable project convention.

When To Use It

  • Use slash commands for user-facing actions.
  • Use subcommands when one feature has several related actions.
  • Use route-based discovery when folders naturally map to command groups.
  • Use manual names when file layout should not affect public command names.

Basic Example

import { Slash } from "@tsundere/discord"

export default Slash.command("ping")
  .description("Check bot latency")

Options Example

export default Slash.command("ban")
  .description("Ban a member")
  .option("user", "target", "Member to ban", { required: true })
  .option("string", "reason", "Reason for the ban", { required: false })
  .permission("BanMembers")
  .guildOnly()

Advanced Example

{
  "commands": {
    "discovery": true,
    "routeBased": false,
    "directory": "./src/commands",
    "groups": {
      "./src/commands/admin": {
        "routeBased": true,
        "groupName": "admin"
      }
    }
  }
}

With this config, `src/commands/admin/ban.yuri` can become `/admin ban` while other folders still use exported command names.

Common Mistakes

  • Registering each command with a bulk PUT route one at a time, which overwrites previous commands.
  • Using global sync while expecting instant development updates.
  • Forgetting `dmPermission(false)` or `guildOnly()` for guild-only moderation commands.
  • Using permission names that do not map to Discord permission bit values.

Performance Notes

Sync commands only when metadata changes. Prefer guild sync during development. Keep command handlers small and push database/API work into services.
