Getting Started

Project Structure

A Tsundere project is a normal Node project with `.yuri` source files, a JSON Tsundere config, npm dependencies, and a generated local runtime folder. Understanding these paths makes troubleshooting much easier.

Starter Layout

my-bot/
  package.json
  package-lock.json
  tsundere.config.json
  tsundere-workspace.yaml
  tsundere-lock.yaml
  .env
  .env.example
  src/
    main.yuri
    commands/
      ping.yuri
  build/
  .tsundere/
    runtime/
      discord/
    runtime-build/
  .yuri-cache/
    discord.commands.json

Not every file exists immediately. `package-lock.json`, `tsundere-workspace.yaml`, and `tsundere-lock.yaml` are written after install. `build` and `.tsundere/runtime-build` are written after build/dev/start. `.yuri-cache/discord.commands.json` is written by command sync and dev mode.

The structure is intentionally close to a normal Node project. You can open `package.json`, inspect scripts, run npm commands, and commit npm lockfiles the same way you would in a JavaScript project. Tsundere adds the `.yuri` source folder, config, generated runtime folders, and package optimizer snapshots around that familiar core.

Config

`tsundere.config.json` is loaded by `src/config.ts`. Defaults exist for `name`, `source`, `outDir`, `target`, `strict`, `sourceMaps`, `runtime`, `plugins`, and diagnostics. If `TSUNDERE_ENV` is set, Tsundere also tries `tsundere.<env>.config.json` after the base config.

{
  "name": "my-bot",
  "source": "src",
  "outDir": "build",
  "target": "typescript",
  "strict": true,
  "sourceMaps": true,
  "commands": {
    "discovery": true,
    "routeBased": true,
    "directory": "./src/commands"
  }
}

Config loading is layered. Tsundere starts with defaults, loads `tsundere.config.json` if it exists, then loads `tsundere.<TSUNDERE_ENV>.config.json` when the environment variable is set. Later layers override earlier ones. Nested diagnostic, Discord, enterprise, and runtime options are merged rather than replaced wholesale where the config loader knows those shapes.

Validation is currently focused on the basics: `name` must be a string, `source` and `outDir` must be directories, and `target` must be `javascript` or `typescript`. That means a typo in a more advanced option might not always fail during config load. When a setting seems ignored, compare it against the TypeScript types and the module that actually consumes it.

Generated Runtime Paths

`build` is not the final runtime folder. First, `.yuri` files compile into `build` as `.ts` or `.js`, depending on `target`. Then `emitNodeRuntime` converts those outputs into runnable JavaScript under `.tsundere/runtime-build`. `tsundere start` runs `.tsundere/runtime-build/main.js`.

The local Discord wrapper lives separately under `.tsundere/runtime/discord`. Its `package.json` declares `@tsundere/discord`, exports `./dist/index.js`, and depends on Discord.js. Your project `package.json` points to it with `file:.tsundere/runtime/discord`.

Build output is safe to delete. The compiler removes and recreates `.tsundere/runtime-build` when emitting runtime output, and it can refresh the bundled Discord runtime dist files from the CLI package. Do not hand-edit generated runtime files as a fix; change the `.yuri` source or the CLI code that emits them.

Package Files

`package.json` remains the source of dependency declarations. npm writes `package-lock.json`. Tsundere reads npm's lockfile to understand installed package names, versions, integrity strings, direct dependency status, and disk locations. It then writes `tsundere-workspace.yaml` and `tsundere-lock.yaml` as an npm-derived snapshot for the optimizer layer.

The global store lives outside the project, normally under `~/.tsundere/store`. Project references tell prune which store entries are still used. This is why `tsundere store prune` can remove unreferenced cache entries without touching your project files. The safety code validates paths before removing or hydrating files so a bad store path cannot escape the intended directory.

Files You Usually Commit

  • `src/**/*.yuri`
  • `package.json` and `package-lock.json`
  • `tsundere.config.json`
  • `tsundere-workspace.yaml` and `tsundere-lock.yaml` if your team wants Tsundere's npm-derived snapshot committed
  • Documentation, examples, and config templates

The starter `.gitignore` excludes `node_modules`, `build`, `.tsundere`, and `.env`. Keep it that way unless you have a specific release artifact reason.

If you are building a deployable archive, create it from source, config, lockfiles, and an install step. Shipping `.tsundere/runtime-build` can be useful for a controlled release bundle, but it should not be the only copy of your bot logic. The source and lockfiles are what let another developer rebuild the project cleanly.
