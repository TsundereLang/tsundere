Getting Started

Linux Setup

Linux support uses Node's platform APIs and POSIX-style paths. The CLI expects Node and npm, uses `sh` for command lookup, and checks executable permissions through `tsundere doctor`.

Prerequisites

Install Node.js and npm from your distribution, NodeSource, fnm, nvm, or another trusted source. For the web installer, `curl` and `unzip` are also required.

node --version
npm --version
command -v curl
command -v unzip

Tsundere's platform detection treats anything that is not `win32` or `darwin` as Linux. That includes many CI Linux environments. Command lookup uses `sh -c "command -v ..."` and normal executable names such as `node`, `npm`, and `npx`; there are no Windows `.cmd` shims on Linux.

If you use nvm, fnm, asdf, or another shell-loaded Node manager, run Tsundere from a shell where the manager has already initialized PATH. Scheduled cron update checks may not load your interactive shell profile, so use a globally reachable Node install if you depend on automation.

Linux Web Installer

`scripts/install-tsundere-linux.sh` downloads the latest GitHub release, extracts it, finds `tsundere-cli.tgz`, checks Node/npm, installs pnpm if missing, installs the CLI globally, optionally installs the VS Code/Cursor/Antigravity extension, and runs `tsundere help`.

sh scripts/install-tsundere-linux.sh

The script has WSL-specific handling. If it detects WSL, it tries to use a Windows-visible temporary folder to avoid Windows npm problems with `\\wsl.localhost` paths.

Like the Windows installer, the Linux script expects the latest release archive to contain `tsundere-cli.tgz`. It downloads a release zip asset when available, otherwise it falls back to the source zipball. A source zipball alone is not enough for a normal install unless the packaged CLI tarball is included inside it.

The script prompts to install `vscode-tsundere-*.vsix` when a VSIX is present and `code`, `cursor`, or `antigravity` is available. This prompt reads from `/dev/tty`, so it expects an interactive terminal for extension installation.

Source Checkout Install

For local development of Tsundere itself, use `scripts/install-linux.sh`. It changes to the repo root, runs npm install, builds, and runs doctor.

sh scripts/install-linux.sh

This is not the same as installing a release. It prepares the current checkout and proves that the built CLI can run doctor. Use it when editing Tsundere's own source. Use the web installer or npm global install path when testing what a normal user receives from a release.

Storage and Permissions

Linux storage uses your home directory:

~/.tsundere/
~/.tsundere/config.json
~/.tsundere/cache/
~/.tsundere/store/
~/.tsundere/logs/

Doctor checks executable permissions for known scripts and the built CLI when present. If permission errors appear, fix ownership or run `chmod +x` on the specific file named by the error.

The permission helper applies mode `755` to non-Windows executables. It cannot fix every environment: files on read-only mounts, package-managed directories owned by root, or restrictive network mounts may still fail. When that happens, move the project to a normal user-owned directory or repair ownership with your system tools.

Opening Docs And Running Updates

On Linux, `tsundere docs` launches `xdg-open` with the local docs HTML file. Minimal server images often do not have `xdg-open` or a desktop browser. In that case, open `docs/local/index.html` manually from the repository or installed package.

Automatic update checks use crontab on Unix-like systems. The updater writes a marked line that runs Node against the installed CLI path. If your Node executable is only available in interactive shells, the cron job may not find it. Check `crontab -l` and run `tsundere updater cron status` when verifying automation.

Common Linux Mistakes

  • Installing Node without npm. Tsundere needs npm for package operations.
  • Running the installer from a filesystem where global npm install cannot write. Use a Node manager or fix npm permissions.
  • Using WSL but launching Windows editor/npm commands against Linux-only paths. Keep the workflow consistently Windows or consistently WSL where possible.

Also avoid running global npm install with `sudo` unless your Node setup explicitly requires it. Mixing root-owned global packages with user-owned projects can make later updates and uninstall steps painful.
