Observability

Tsundere projects should be easy to inspect in development and production. Observability covers structured logs, metrics, traces, dashboards, alerts, and runtime health.

Overview

A production bot needs more than console output. You need to know when commands are slow, when Discord REST calls are rate limited, when shards reconnect, when memory grows, and when a deployment starts failing.

Tsundere's observability direction is built around structured events that can flow into Prometheus, Grafana, Loki, OpenTelemetry, Sentry, Datadog, or a custom webhook.

Why It Exists

Discord bots often fail quietly. A missing intent, stale command sync, bad permission, or API rate limit can look like a user problem from the outside. Observability turns those invisible failures into searchable records and measurable signals.

When To Use It

  • Use pretty logs while developing locally.
  • Use JSON logs in production, Docker, systemd, and hosted workers.
  • Use metrics when you need uptime, latency, memory, CPU, command counts, or REST rate limit dashboards.
  • Use traces when one user action touches Discord, a database, a queue, and an external API.

Structured Logging Shape

{
  "level": "info",
  "time": "2026-06-06T12:00:00.000Z",
  "service": "tsundere-bot",
  "shard": 0,
  "requestId": "cmd_01HX",
  "event": "command.completed",
  "command": "ping",
  "guildId": "123",
  "latencyMs": 42
}

Use stable field names so logs remain searchable. Avoid building logs from only human sentences when machines also need to read them.

Development Logging

import { logger } from "@tsundere/logger"

const log = logger({
  service: "community-bot",
  format: "pretty",
  redact: ["DISCORD_TOKEN", "GITHUB_TOKEN"]
})

log.info("Bot online", {
  user: client.user.tag,
  shard: client.shardId ?? 0
})

The logger package is a planned official package. Until it lands, follow the same shape with your preferred npm logger.

Metrics To Track

Runtime

Uptime, memory, CPU, event loop lag, shard status, reconnects, cache size, gateway latency.

Discord

Command count, interaction latency, REST status, route buckets, rate limits, failed replies, missing permissions.

Compiler

Build duration, file count, cache hits, diagnostic count, generated output size.

Package Manager

Install duration, cache hits, reused packages, store validation failures, lockfile changes.

Prometheus Example

# HELP tsundere_command_duration_ms Command execution duration.
# TYPE tsundere_command_duration_ms histogram
tsundere_command_duration_ms_bucket{command="ping",le="50"} 18
tsundere_command_duration_ms_bucket{command="ping",le="100"} 24

# HELP tsundere_discord_rest_requests_total Discord REST requests.
# TYPE tsundere_discord_rest_requests_total counter
tsundere_discord_rest_requests_total{method="PUT",route="/commands",status="200"} 1

Grafana And Loki

Send JSON logs to Loki and metrics to Prometheus, then build Grafana panels for command latency, bot uptime, shard reconnects, rate limit pressure, memory, CPU, and deployment version.

{service="tsundere-bot", event="command.failed"}
| json
| command="ban"

Common Mistakes

  • Do not log Discord tokens, OAuth secrets, webhook URLs, or database passwords.
  • Do not log every message create event in busy servers unless there is a clear reason.
  • Do not rely only on Discord channel logs for production health.
  • Do not attach high-cardinality values like raw message IDs to every metric label.

Performance Notes

Prefer batched metrics and non-blocking exports. Logging should not slow down command responses. Redact secrets before logs leave the process, not after they arrive in a remote system.
