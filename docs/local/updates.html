Updates

Updates in Tsundere are split across three real workflows: project package updates, CLI self-updates from GitHub releases, and release bundle publishing for maintainers. This page explains the difference so you do not use a release command when you meant to update a bot dependency.

The Short Version

Use `tsundere update` for dependencies inside the current project. Use `tsundere updater self --yes` to update the installed Tsundere CLI. Use `npm run dist:release` only when building a Tsundere release bundle from this repository.

tsundere update discord.js
tsundere updater self --dry-run
tsundere updater self --yes
npm run dist:release

Project Package Updates

Project package updates go through the package optimizer in `src/package-optimizer.ts`. npm still resolves packages and owns `package-lock.json`. Tsundere wraps that npm operation, repairs the local Discord runtime when needed, records install metrics, harvests packages into the store, and writes `tsundere-workspace.yaml` plus `tsundere-lock.yaml` after a successful install/update.

tsundere add zod
tsundere remove zod
tsundere update discord.js
tsundere install

This does not update the global Tsundere CLI. It changes dependencies for the project in your current working directory. If `tsundere update` fails, compare it with plain npm behavior only as a diagnostic step. When plain npm succeeds but Tsundere fails, the issue is likely in store hydration, store harvesting, local runtime repair, or Tsundere YAML snapshot generation.

CLI Release Checks

The updater implementation in `src/updater.ts` reads `https://api.github.com/repos/TsundereLang/tsundere/releases/latest` by default. The repo can be changed with `TSUNDERE_UPDATE_REPO`. Release tags are normalized by trimming a leading `v`, and version comparison is numeric dot comparison.

tsundere version
tsundere updater check
tsundere updater info

If GitHub cannot be reached, the updater reports that it could not read the latest release. It does not guess and it does not install anything. Network failures, GitHub rate limits, private repository auth problems, and an incorrect repo value can all look similar from the CLI.

Self Update Flow

Self-update selects a CLI tarball asset from the latest release. It prefers a versioned file like `tsundere-cli-0.1.1.tgz`, then `tsundere-cli.tgz`, then any `tsundere-cli*.tgz` asset. Without `--yes`, the command reports the available update and exits without installing. With `--dry-run`, it verifies selection without changing the machine.

tsundere updater self --dry-run
tsundere updater self --yes
tsundere updater self --yes --force

With `--yes`, the updater downloads the selected tarball to a temporary `tsundere-update-*` folder and runs npm global install on that package. The temporary tarball is removed afterward. If npm global permissions are broken, the updater can find the release and still fail during install.

Automatic Update Checks

Automation checks for updates; it does not silently install them. On Windows, `tsundere updater cron install` creates a Scheduled Task named `Tsundere Daily Update Check`. On Unix-like systems, it edits the current user's crontab with a marked `tsundere updater check` line.

tsundere updater cron install
tsundere updater cron status
tsundere updater cron remove
tsundere updater cron install --time 09:30

Security update notices are separate. Before commands such as `dev`, `start`, `build`, `install`, `add`, and `update`, Tsundere can check whether the latest release text looks security-related. That result is cached for 24 hours in `~/.tsundere/security-update-cache.json`. Set `TSUNDERE_NO_UPDATE_NOTICE=1` to suppress the notice in automation.

Release Bundle Builds

Maintainers build release artifacts with `scripts/build-distribution.ps1`, exposed as `npm run dist:release`. The script installs dependencies using a local `.npm-cache`, runs the TypeScript build, packs the CLI, packs `packages/discord`, packages the VS Code extension, copies installer scripts, creates installer aliases, calculates checksums, writes `release-manifest.json`, and creates a `Tsundere-v<version>` bundle archive.

npm run dist:release
npm run dist:release -- -Version 0.1.1

The generated `release/` folder can contain `tsundere-cli.tgz`, `tsundere-discord.tgz`, versioned tarballs, `install-tsundere.ps1`, `installer.bat`, `install-tsundere-windows.ps1`, `install-tsundere-linux.sh`, `update-tsundere.ps1`, a VSIX, optional Electron installer EXE, checksums, docs, and the final `.rar` or `.zip` bundle.

Publishing To GitHub

`scripts/create-github-release.ps1` publishes a GitHub release. It can prompt for the repo, version, draft status, and whether to rebuild the bundle first. It writes `release/RELEASE_NOTES.md` using `updates.md`, then uses GitHub CLI when available. If `gh` is not installed, it falls back to GitHub API calls using `GH_TOKEN`, `GITHUB_TOKEN`, or Git Credential Manager credentials.

powershell -ExecutionPolicy Bypass -File scripts/create-github-release.ps1
powershell -ExecutionPolicy Bypass -File scripts/create-github-release.ps1 -Version 0.1.1 -DryRun

The publish script uploads the bundle archive, not every release file individually. If you need `tsundere updater self` to install directly from GitHub release assets, make sure the release also exposes a matching CLI tarball asset or adjust the release layout before recommending self-update. The updater only knows how to install `tsundere-cli*.tgz` assets.

Reading `updates.md`

`updates.md` is Discord-friendly project update copy. It is useful for communicating what changed in a release cycle, but it is not executable state. A bullet in `updates.md` does not prove an installer asset exists, and a release asset does not prove the docs have been updated. Treat release notes, source files, tests, and assets as separate evidence.

When a release mentions runtime, package, installer, or GitBot changes, verify the code paths that matter to your project: `tsundere doctor`, `tsundere install`, `tsundere build`, `tsundere commands sync`, `tsundere dev`, and any Discord flows your bot relies on.

Common Update Mistakes

  • Using `tsundere updater self --yes` when you meant to update `discord.js` in a project.
  • Publishing a release bundle without a `tsundere-cli*.tgz` asset and then expecting self-update to work.
  • Running release scripts from a project folder instead of the Tsundere CLI repository root.
  • Assuming scheduled update checks install updates automatically. They only run `updater check`.
  • Forgetting that installed local docs only update when the CLI package itself is updated.
