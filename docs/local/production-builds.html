Getting Started

Production Builds

Production mode is a two-step idea: build the `.yuri` source into runtime output, then run the generated JavaScript through the Tsundere launcher.

Build

tsundere build

`buildProject` scans the configured `source` directory for `.yuri` files. It writes `.ts` or `.js` files into `outDir` based on the `target` config. If source maps are enabled, it also writes `.map` files.

After that, Tsundere refreshes `.tsundere/runtime/discord` from the bundled CLI output and emits runnable JavaScript into `.tsundere/runtime-build`. If the config target is TypeScript, it uses TypeScript's `transpileModule` to turn the intermediate `.ts` into ESM JavaScript.

The build command exits with code `1` if compiler diagnostics include errors. Warnings are printed unless diagnostics config disables them. If no `.yuri` files exist under `source`, the build warns with the resolved source path instead of pretending the project is fine. That warning usually means `source` is wrong or the project was created without source files.

Build output is generated. It is fine to delete `build` and `.tsundere/runtime-build` before a clean build. The runtime emitter removes `.tsundere/runtime-build` before rewriting it, using the path safety helpers so cleanup stays inside the project. If cleanup fails because a process is still using the files, stop the running bot and build again.

Start

tsundere start

`tsundere start` checks for `.tsundere/runtime-build/main.js`. If it is missing, the CLI builds first. Then it spawns Node on that runtime entry and waits for the process to exit.

For deployment, ship the project files needed by npm and Tsundere, run `tsundere install`, set `.env` or environment variables on the host, run `tsundere build`, and launch with `tsundere start` or a process manager that calls that command.

The runtime process inherits your environment plus values parsed from `.env`. In a hosted environment, you can skip the `.env` file and set real environment variables instead. If both exist, `.env` values are merged into the spawned process environment by the launcher, so keep deployment secrets consistent and avoid stale local files on servers.

Use the same command in your process manager that you test locally. For example, a simple process entry can call `tsundere start` from the project root. Starting `.tsundere/runtime-build/main.js` directly can work, but you lose Tsundere's missing-build check, `.env` load, runtime branding, and future startup checks.

Protected Builds

tsundere build --protect standard

The CLI accepts `--protect` with `standard`, `advanced`, or `maximum`. The current protection pipeline lives in `src/compiler/protect.ts`; it can add generated guard code and write `.tsundere/runtime-build/tsundere-protect.json` with fingerprint metadata. Treat this as an early protection feature, not a substitute for real licensing, secrets management, or server-side controls.

tsundere fingerprint inspect

You can also pass `--seed` to make protection output more repeatable for a given seed. Without a seed, protection metadata records `auto`. The fingerprint command reads the generated metadata and reports protected runtime files. If no protected build exists, inspect has nothing meaningful to report.

A Practical Release Checklist

  1. Run `tsundere install` so npm and Tsundere package snapshots are current.
  2. Run `tsundere build` and fix any diagnostics.
  3. Run `tsundere start` once from the project root.
  4. Confirm `.env` or deployment environment variables are present on the target machine.
  5. Commit source, config, package files, and lockfiles. Do not commit secrets.

For the Tsundere CLI repository itself, release packaging is separate from project builds. The repo script `npm run dist:release` calls the PowerShell distribution builder. That creates installer-oriented artifacts for the CLI, not a production build of your bot project.

Common Mistakes

  • Running `node build/main.ts` directly. Tsundere's runtime entry is `.tsundere/runtime-build/main.js`.
  • Deleting `.tsundere/runtime/discord` and then starting without reinstalling. Run `tsundere runtime install` or `tsundere install`.
  • Expecting `--protect` to hide tokens. Secrets belong in environment variables, never in source.
