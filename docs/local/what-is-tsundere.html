Getting Started

What Is Tsundere?

Tsundere is a Discord-focused `.yuri` language toolchain and CLI. It compiles `.yuri` files into JavaScript or TypeScript, bundles a local Discord wrapper, and gives projects a workflow around npm, command discovery, local docs, runtime output, and update checks.

What the Repo Actually Contains

The main package is `@tsundere/cli`. Its executable is `dist/cli.js`, built from `src/cli.ts`. The compiler lives in `src/compiler`, the Discord wrapper source lives in `src/discord`, platform helpers live in `src/platform`, package optimization lives in `src/package-optimizer.ts`, and local docs live in `docs/local`.

There are also package folders for the generated Discord runtime, YuriLS, VS Code support, and testing helpers: `packages/discord`, `packages/yurils`, `packages/vscode-tsundere`, and `packages/testing`. The project is moving toward a cleaner package split, but today the CLI still imports many modules from `src` directly.

The repository is not just one feature. It includes a CLI, compiler, runtime bridge, package manager layer, updater, installers, local docs, VS Code extension package, Discord examples, and tests. That breadth is why the Getting Started docs focus on workflows instead of only syntax. A beginner needs to know where source ends and generated runtime begins.

What Tsundere Does Well Today

  • Creates starter projects with `tsundere create my-bot --template discord`.
  • Compiles `.yuri` from `src` into `build`, then emits runnable JavaScript into `.tsundere/runtime-build`.
  • Installs `@tsundere/discord` as a local file dependency from `.tsundere/runtime/discord`.
  • Wraps npm install/update/remove while preserving `package-lock.json` and normal Node resolution.
  • Writes Tsundere YAML snapshots: `tsundere-workspace.yaml` and `tsundere-lock.yaml`.
  • Discovers commands under `src/commands` and writes `.yuri-cache/discord.commands.json`.
  • Provides `tsundere doctor`, `tsundere docs`, `tsundere updater`, `tsundere metrics`, and `tsundere stress runtime`.

The most mature path today is a local Discord project. The starter template, runtime bridge, command discovery, interaction helpers, and tests are all oriented around that. The package optimizer and Linux/platform work are useful outside Discord too, but the docs should be honest that Discord bot development is the primary target right now.

What Is Still Experimental

The `.yuri` compiler currently uses a set of targeted syntax-lowering passes. It supports useful TypeScript-like and Discord-focused patterns, but it is not yet a complete language parser with full type checking. If you write ordinary JavaScript-like code, imports, functions, object calls, and Discord wrappers are the safest path.

The distributed runtime work is also a foundation. Commands like `tsundere inspect`, `tsundere metrics doctor`, and `tsundere stress runtime` exist. Local IPC, event, cache, task, Prometheus, and Grafana helpers exist. Fully automated multi-machine Discord clustering is not finished yet, so production bots should still be tested carefully.

Some docs pages outside Getting Started describe roadmap or planned systems. Treat those as direction, not guaranteed current behavior. When you need the current truth, check the CLI help, the source modules, and tests. These Getting Started pages are written from those implementation files.

Where Tsundere Fits In A Node Project

A Tsundere project still has `package.json`, `package-lock.json`, `node_modules`, npm scripts, and normal Node module resolution. Tsundere adds `.yuri` source and an opinionated CLI around building and running that source. The optimizer can reuse package files across projects, but it does not replace npm and does not ask you to abandon npm lockfiles.

For a Discord bot, Tsundere also adds `@tsundere/discord`, which is installed from a project-local file dependency. That package is generated by the CLI under `.tsundere/runtime/discord` and backed by the CLI's built Discord wrapper files. If that folder is missing, the bot will not resolve Tsundere's Discord import until you repair it with `tsundere runtime install` or `tsundere install`.

Common Misunderstandings

Tsundere is not pnpm. `tsundere install` still uses npm and preserves npm lockfile behavior. The store layer is an optimization around npm, not a replacement.

Tsundere is not a complete bot framework. It does not ship economy, tickets, moderation packs, or dashboards as prebuilt systems. It gives you a cleaner way to build your own.

Tsundere is not separate from Node. The runtime is a Node-based runtime with Tsundere branding and generated output. You still need Node.js and npm.
