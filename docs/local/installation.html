Getting Started

Installation

Tsundere installs as a Node CLI. The command exposed by `package.json` is `tsundere`, pointing to `./dist/cli.js`, so a release install must include built `dist` output and a packaged CLI tarball.

Prerequisites

The CLI doctor checks for Node and npm through `src/platform/index.ts`. npm is required because `tsundere install`, `tsundere add`, `tsundere remove`, `tsundere update`, and the updater self-install all call npm-compatible commands. pnpm is used by the repository workflow, but project installs are npm-first.

node --version
npm --version

Use a modern Node version that can run ES modules and the built TypeScript output. The package currently depends on TypeScript and Discord.js, and the runtime starts projects with `process.execPath`, which means the same Node executable that launched Tsundere is used to launch your generated bot. If your terminal has multiple Node installs, check the version from the same shell where you plan to run `tsundere`.

The install should also be run from a normal user account, not from a system directory. Tsundere writes project files in the folder you choose, and global storage under your home directory. Windows, Linux, and macOS all use platform helpers rather than hardcoded paths, but permission problems still happen when a project sits under Program Files, a protected system path, or a locked sync folder.

Install From a Release Bundle

The web installer scripts fetch the latest GitHub release from `https://api.github.com/repos/TsundereLang/tsundere/releases/latest`, download a zip asset or source zipball, find `tsundere-cli.tgz`, and install that tarball globally with npm. The Windows script is `scripts/install-tsundere-windows.ps1`; the Linux script is `scripts/install-tsundere-linux.sh`.

powershell -ExecutionPolicy Bypass -File scripts/install-tsundere-windows.ps1

sh scripts/install-tsundere-linux.sh

Those scripts may also offer to install the VS Code/Cursor extension if a `vscode-tsundere-*.vsix` file is present in the release bundle and an editor command is available.

A release bundle install is different from cloning the repository. The installer expects release artifacts to already exist: a CLI tarball named like `tsundere-cli.tgz` and, optionally, a VS Code extension package. If you run the installer against a source-only download before release artifacts are attached, the script can download the code but still fail to find the installable CLI package. In that case, use the source checkout steps below or rebuild the release package first.

Install From a Source Checkout

When working directly in this repository, build the CLI first. `npm run build` runs TypeScript and then `scripts/prepare-discord-runtime.mjs`, which copies `dist/discord` into `packages/discord/dist` so the local runtime package is ready.

npm install
npm run build
node dist/cli.js doctor

To install globally from a generated CLI tarball, use the release workflow. Do not point beginners at `node dist/cli.js` as a permanent install. That command is fine for local repo development, but normal users should get the `tsundere` binary through npm global install or a release installer.

The source checkout route is useful for contributors because it keeps the repo editable. `npm install` creates the repo `node_modules` and `package-lock.json` state. `npm run build` compiles `src/**/*.ts` to `dist` and prepares the bundled Discord runtime. `node dist/cli.js doctor` then proves that the built CLI can load config defaults, detect the platform, create the Tsundere home folders, find Node and npm, and inspect the package optimizer store.

If you want the command name while working from source, use your normal npm linking workflow after the build. Remember that linking is a development convenience. If a tester reports an issue with the installer, reproduce with the release bundle, not with a linked checkout, because those paths exercise different code.

Verify the Install

tsundere version
tsundere doctor
tsundere docs

`tsundere version` prints the installed Tsundere version and the configured GitHub release repo. `tsundere doctor` loads config defaults, checks platform paths, verifies Node and npm, and inspects the package optimizer store. `tsundere docs` opens `docs/local/index.html` from the installed CLI package.

A healthy doctor run should show the project name, source directory, output directory, target, strict mode, platform label, Tsundere home, config/cache/log paths, and runtime checks. On Linux it also checks executable permissions for scripts that need them. If the doctor command exits with a non-zero code, read the package optimizer lines near the end; invalid store entries or missing runtime tools are reported there.

Where Installation Writes Files

The CLI package itself is installed by npm wherever your npm global prefix points. Tsundere's own user-level data is separate. The platform module builds paths under your home directory, including `~/.tsundere/config.json`, `~/.tsundere/cache`, `~/.tsundere/store`, and `~/.tsundere/logs`. The package optimizer defaults its global store to that same Tsundere home area unless `storePath` is set in config.

Project-level generated files stay inside the project. A Discord project can contain `.tsundere/runtime/discord`, `.tsundere/runtime-build`, `.yuri-cache/discord.commands.json`, `build`, `node_modules`, npm lock files, and Tsundere YAML package snapshots. This split matters for cleanup: `tsundere cache clean` clears the global store, while deleting `build` or `.tsundere/runtime-build` only affects one project.

Common Mistakes

  • Installing from a source zip that does not contain `tsundere-cli.tgz`. The installer expects the release bundle to include the CLI tarball.
  • Running `tsundere dev` before `tsundere install`. New projects need dependencies and the local Discord runtime package installed.
  • Using a terminal that cannot find global npm binaries. Reopen the terminal after global install, or check your PATH.
