Language Fundamentals

Tsundere uses `.yuri` files with familiar JavaScript and TypeScript structure, then adds stricter compiler checks, Discord-aware diagnostics, local runtime output, and editor metadata.

Overview

A `.yuri` file is source code that the Tsundere compiler lowers into JavaScript or TypeScript. The language keeps imports, objects, arrays, classes, async functions, and builder chains familiar so npm packages work naturally.

The main difference is that Tsundere treats Discord development as a first-class workflow. It can understand command files, interaction checks, component builders, embed limits, intents, permissions, and generated Discord metadata.

Why It Exists

Discord.js gives developers a powerful API, but many projects repeat the same setup: command loaders, command sync scripts, interaction switches, custom ID parsing, embed validation, and event intent mistakes. Tsundere keeps the API flexible while making the repeated parts easier to reason about.

When To Use It

  • Use `.yuri` for Discord bot code, command modules, services, and runtime entry points.
  • Use npm packages directly when you need databases, HTTP clients, utility libraries, or build tools.
  • Use TypeScript output when you want another tool to inspect generated types.
  • Use JavaScript runtime output when you want the fastest path to `tsundere start`.

Variables And Types

const name: string = "Yuri"
let retries = 3
let enabled: boolean = true

type BotConfig = {
  token: string
  guildId?: string
  debug: boolean
}

Prefer `const` when a binding does not change. Add explicit types at module boundaries, exported APIs, handlers, and configuration objects. Let inference handle short local values.

Functions

export async function replyWithLatency(interaction, client) {
  await interaction.reply({
    content: `Latency: ${client.ping}ms`,
    ephemeral: true
  })
}

Async functions compile to normal JavaScript async functions. Use `await` for Discord API calls and REST operations so rate limit behavior remains predictable.

Modules And Imports

import { Client, Intents, Slash } from "@tsundere/discord"
import mysql from "mysql2"
import { loadWarnings } from "./services/warnings"

export default Slash.command("warnings")
  .description("View member warnings")

Tsundere preserves package imports. If Node can resolve the package after `tsundere install`, Tsundere can compile the import. Local `.yuri` imports are rewritten to runnable JavaScript paths in runtime output.

Discord-Native Layout

src/
  main.yuri
  commands/
    ping.yuri
    admin/
      ban.yuri
  events/
    ready.yuri
  services/
    warnings.yuri

Keep `main.yuri` small. Put commands in `src/commands`, reusable logic in `src/services`, and event handlers in their own files when the bot grows.

Advanced Example

import { Component, Button, Row } from "@tsundere/discord"

const ConfirmAction = Component.define<{
  actionId: string
  actorId: string
}>("confirm")

export function actionRow(actionId, actorId) {
  return Row.of(
    Button.success(ConfirmAction.create({ actionId, actorId })).label("Confirm"),
    Button.secondary("cancel").label("Cancel")
  )
}

Typed components keep custom IDs from becoming string parsing projects. The compiler and language server can use the component definition to guide handlers and hover docs.

Common Mistakes

  • Do not treat Tsundere as a prebuilt bot framework. It gives wrappers and tools, not finished systems.
  • Do not manually run `node build/main.ts`. Use `tsundere start` after `tsundere build`.
  • Do not ignore intent warnings. A bot can look online while handlers never receive events.
  • Do not store secrets in source files. Use `.env`, deployment secrets, or managed runtime configuration.

Performance Notes

Tsundere compiles to normal Node-compatible modules. Keep hot paths simple, avoid excessive REST calls in event handlers, cache Discord lookups where appropriate, and use `tsundere dev` for rapid local rebuilds.
