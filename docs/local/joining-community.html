Getting Started

Joining The Community

Tsundere is early and moving fast, so the community is where rough edges, install problems, Discord runtime behavior, and docs gaps should be reported clearly.

Main Links

  • GitHub organization: `https://github.com/TsundereLang`
  • Main repo: `https://github.com/TsundereLang/tsundere`
  • Discord invite from the README: `https://discord.gg/Gpxj5xVXBZ`
  • Local docs entry: `docs/local/index.html`

Use GitHub for issues that need history, reproduction steps, code review, or release tracking. Use Discord for quick questions, install help, and discussion while a feature is still being shaped. If a Discord conversation uncovers a real bug, turn the final repro into a GitHub issue so it does not disappear into chat history.

How To Ask For Help

When reporting a problem, include the command you ran, your OS, Node version, Tsundere version, and the relevant output. For project issues, include the shape of `tsundere.config.json`, whether `@tsundere/discord` exists under `.tsundere/runtime/discord`, and whether `tsundere doctor` passes.

tsundere version
tsundere doctor
node --version
npm --version

For install problems, also say whether you used the Windows installer, Linux installer, source checkout, npm global install, or updater self-install. Those paths use different code. For package problems, include whether plain `npm install` works and whether `tsundere install` fails before or after npm runs.

For Discord runtime problems, include the event or interaction that failed, the relevant `.yuri` source, and the exact runtime stack trace. If the stack trace points into `.tsundere/runtime-build`, include the source file that produced it too. Generated output is useful evidence, but source is where the fix usually belongs.

For docs problems, include the page name and whether you are reading the installed local docs, the repository docs, or GitHub-rendered files. The docs are shipped with releases, so a fixed page in the repository may not be present on an older installed CLI.

What Not To Share

Never paste your real `DISCORD_TOKEN`, GitHub token, npm token, `.env` contents, private bot secrets, or release credentials. The starter already ignores `.env`; keep it that way.

If you need to show config, replace sensitive values before posting. Prefer small repro projects over screenshots of your whole desktop. A clean report with one command, one error, and a tiny source file gets fixed faster than a wall of unrelated logs.

Contributing Back

If you are working in the repo, read `CONTRIBUTING.md`, `SECURITY.md`, and `docs/ARCHITECTURE.md`. The project is transitioning toward clearer packages, so changes that reduce coupling and improve tests are especially useful.

Good contribution areas include docs that match source behavior, tests around package optimizer edge cases, Discord runtime compatibility, Linux and Windows path handling, updater release layout checks, and examples that compile without hidden setup. Before opening a PR, run the relevant build and tests and mention what you verified.

Keep changes focused. A fix to `member.roles.add` should come with a runtime test or a small reproduction, not unrelated formatting across the repo. A docs rewrite should cite the commands and files it describes. A release fix should prove the generated asset layout works with the installer and updater paths.

A Useful Support Template

OS:
Tsundere version:
Node version:
npm version:
Command:
Expected:
Actual:
Relevant files:
What I already tried:

That small structure gives maintainers enough to reproduce the issue without guessing. If the issue is security-sensitive, do not post it publicly; follow the security policy instead.

The best reports usually include one minimal command sequence. For example: create a fresh starter, run install, set a dummy token or mock gateway when possible, then run the failing command. If a bug only happens in a private production bot, try to reduce it to a small `.yuri` file before asking others to inspect the entire application.
