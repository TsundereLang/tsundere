Overview

Ready documents the Discord gateway event `ready`. Tsundere exposes it through `client.on("ready", ...)` and uses generated Discord metadata so editor completions, callback parameters, and intent warnings stay close to Discord.js behavior.

Purpose And Use Cases

Initialize services, sync commands, schedule background jobs, and announce startup once the gateway is ready. Required intent guidance: Guilds. Use this page when deciding whether an event belongs in your bot, which partials are needed, and how to avoid noisy production listeners.

Architecture Or Design Notes

Gateway architecture: Discord.js receives the raw gateway event, @tsundere/discord maps it into a Tsundere-friendly typed object, YuriLS uses metadata for completions, and diagnostics warn when required intents or partials are missing. Callback signature: `()`.

Beginner Example

import { Client, Intents } from "@tsundere/discord"

const client = new Client({
  token: env.DISCORD_TOKEN,
  intents: [Intents.Guilds]
})

client.on("ready", async () => {
  log("ready received")
})

client.login()

Advanced Example

client.on("ready", async () => {
  try {
    await metrics.counter("discord_events_total").inc({ event: "ready" })
    await audit.write({ event: "ready", receivedAt: new Date().toISOString() })
  } catch err {
    log("Event handler failed: " + err.message)
  }
})

Reference And API Details

Event name: `ready`. Callback parameters: `()`. Required or recommended intents: Guilds. Use partials when Discord can emit incomplete cached data, especially for message delete and update flows.

Best Practices

  • Register only the ready listener when the bot actually needs it.
  • Enable the smallest required intent set: Guilds.
  • Keep event handlers thin and move business logic into services.
  • Log failures with enough context to debug without leaking tokens or private data.

Common Mistakes

  • Forgetting the required gateway intent or privileged intent toggle.
  • Assuming uncached messages or members always include every field.
  • Doing heavy database work directly inside the event callback.
  • Logging message content or user data without a clear moderation need.

Troubleshooting

  • Confirm the bot is listening for `ready` and that the gateway connection reached ready.
  • Verify required intents: Guilds.
  • Check Discord developer portal privileged intent settings when member, presence, or content data is missing.
  • Use structured logs to confirm whether the event did not fire or the handler failed.
