Overview

GitHub Actions explains GitHub workflow for production Tsundere apps. It focuses on reproducible installs, clear runtime startup, environment safety, and recovery when deployment goes sideways.

Purpose And Use Cases

Use this page when preparing a release, debugging a failed host, or documenting how a project starts after `tsundere build`. It connects the local development workflow to production operators and CI logs.

Architecture Or Design Notes

Deployment architecture starts with `tsundere install`, produces artifacts with `tsundere build`, then runs generated runtime output with `tsundere start`. Production hosts should provide secrets through environment variables and keep generated files separate from source edits.

Beginner Example

name: build
on: [push]
jobs:
  test:
    runs-on: ubuntu-latest

Advanced Example

steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
    with:
      node-version: 24
  - run: npm ci
  - run: npm run build
  - run: npm test

Reference And API Details

Reference notes: the production path is `tsundere install`, `tsundere build`, then `tsundere start`. Keep `.env` out of Git, confirm `@tsundere/discord` resolves locally, and validate host-specific configuration with `tsundere doctor`.

Best Practices

  • Build once, then run the generated runtime artifact consistently.
  • Store secrets in environment variables or platform secret managers.
  • Run health checks before switching traffic or declaring a deploy complete.
  • Document rollback steps before a production incident happens.

Common Mistakes

  • Deploying before running `tsundere build` and tests locally or in CI.
  • Committing `.env` files or bot tokens.
  • Starting Node manually with the wrong generated file instead of using `tsundere start`.
  • Forgetting privileged Discord intent toggles in the developer portal.

Troubleshooting

  • Run `tsundere doctor` on the target machine or deployment image.
  • Verify `DISCORD_TOKEN`, guild IDs, command sync settings, and runtime package resolution.
  • Check process manager logs before editing source code.
  • If startup works locally but not on the host, compare Node versions and environment variables.
