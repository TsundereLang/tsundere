Discord Events

The current runtime event surface is defined by `DiscordEvents` in `src/discord/types.ts` and emitted by `src/discord/client.ts`. Use this page as the runtime truth, not a broad Discord.js event list.

Basic Listener Shape

client.on("messageCreate", async (message) => {
  if (message.author.bot) {
    return
  }

  if (message.content == "!ping") {
    await message.reply("pong")
  }
})

`client.on` and `client.once` are typed from the runtime `DiscordEvents` interface. They register listeners on Tsundere's internal `EventEmitter`. In real gateway mode, Discord.js events are mapped into Tsundere runtime objects and emitted through that internal emitter.

Use `once` for startup tasks that should happen a single time, such as logging the bot tag or registering commands. Use `on` for events that repeat, such as messages, interactions, guild member changes, and voice state updates.

Intent Notes

Tsundere passes your numeric intent array straight into Discord.js. `Guilds` is the base for most guild and interaction work. `GuildMessages` is needed for guild message events. `MessageContent` is needed when your bot reads message text. `GuildMembers` is needed for member join and leave flows. `GuildVoiceStates` is needed for voice state changes.

Intent warnings are generated from source patterns in strict mode. For example, `messageCreate` without `GuildMessages` can trigger `DISCORD002`, and reading `message.content` without `MessageContent` can trigger `DISCORD003`. These warnings are static hints, not Discord API guarantees. A bot can still fail at runtime if the intent is not enabled in the Discord Developer Portal.

Partial Data

Runtime mappings normalize Discord.js objects into Tsundere `User`, `Guild`, `Channel`, `Member`, `Message`, and `VoiceState` shapes. Some fields can be missing when Discord sends partial data or when the bot does not have the required intent or permission. Fetch through `client.guilds`, `client.channels`, or `guild.members` when your handler needs fresh state.

The current mapper intentionally keeps objects small. A mapped `Message` includes IDs, author, content, channel, and an optional `reply` helper. A mapped `Member` includes `id`, `user`, `guildId`, and a roles helper. If you need a Discord.js field that is not exposed by the Tsundere shape yet, fetch through the runtime managers or extend the wrapper deliberately instead of assuming every Discord.js property is present.

Error Handling

The gateway error event is special. If your code has an `error` listener, Tsundere emits the error to your listener. If no `error` listener exists, the runtime prints `Tsundere Discord gateway error: ...` to stderr instead of crashing through an unhandled EventEmitter error.

client.on("error", (error) => {
  log(`Discord gateway error: ${error.message}`)
})

Keep error handlers small. Log the error, increment metrics, or notify an admin channel if that is safe. Do not put heavy recovery logic inside every gateway error; let dev mode or your process manager restart the bot when the process truly exits.

Event Troubleshooting

  • If an event never fires, check both code intents and Developer Portal privileged intent settings.
  • If TypeScript rejects an event name, check whether it exists in `DiscordEvents` for this runtime version.
  • If `message.content` is blank, treat it as an intent problem before rewriting message logic.
  • If a mapped object lacks a property, fetch a fresh object or extend the runtime wrapper.
  • If dev mode keeps restarting, read the first stack trace before the `Waiting for changes...` line.
