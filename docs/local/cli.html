CLI and Runtime

The CLI in `src/cli.ts` is the control surface for Tsundere projects. It creates projects, wraps npm, compiles `.yuri`, emits the Node runtime, opens local docs, manages updater checks, and exposes the distributed runtime foundation. You should not need to run generated files manually for normal work.

The Main Workflow Commands

Most users live on a small set of commands. `create` writes a starter, `install` runs npm through the optimizer, `dev` builds and watches, `build` emits runtime output, and `start` runs that output. These are the commands to learn first.

tsundere create my-bot --template discord
cd my-bot
tsundere install
tsundere dev
tsundere build
tsundere start

`dev`, `start`, `build`, `install`, `add`, and `update` may also show a cached security update notice before running. Set `TSUNDERE_NO_UPDATE_NOTICE=1` in automation if that notice is not desired.

Runtime Flow

`tsundere build` loads config, scans the configured `source` directory, compiles `.yuri` files into `build/`, refreshes `.tsundere/runtime/discord`, and emits runnable JavaScript into `.tsundere/runtime-build/`.

`tsundere start` runs `.tsundere/runtime-build/main.js`. If the runtime entry is missing, it builds first.

`tsundere dev` writes command metadata, builds, starts the runtime, watches `.yuri` files, and restarts the child process when source changes.

The child runtime is still Node. Tsundere launches it with `process.execPath`, forwards stdout, rewrites Node version footers in stderr to say `Tsundere Runtime <version>`, and merges values from `.env` into the child process environment.

Distributed Runtime Foundation

The distributed runtime commands exist today, but they are a foundation rather than a full multi-machine orchestrator. `createRuntimePlan` calculates workers and shards from config, CPU count, simulated shard settings, and environment hints. `DistributedRuntime` provides a local event bus, global event emitter, in-memory cache, task registry, metric snapshots, Prometheus formatting, and Grafana dashboard export.

tsundere inspect
tsundere metrics doctor
tsundere metrics export-grafana ./grafana/tsundere-dashboard.json
tsundere reload

`inspect` prints the current plan, memory, CPU, uptime, cache stats, and metrics endpoint. `reload` does not perform zero-downtime process handoff yet; it reports the staged plan for graceful worker replacement. Use these commands for visibility and testing, not as proof that production clustering is complete.

Runtime Stress Testing

tsundere stress runtime
tsundere stress runtime --heavy
tsundere stress runtime --iterations 100000 --shards 128 --cache-entries 50000 --tasks 500 --metrics-samples 500 --payload-bytes 1024
tsundere stress runtime --json
tsundere stress system
tsundere stress system --heavy
tsundere stress system --packages 150 --yuri-files 100 --commands 50 --build-repeats 3 --hydrate-repeats 3 --json

The stress runner measures local distributed runtime behavior: scale planning, simulated shards, IPC broadcast delivery, global event delivery, cache set/get/miss paths, singleton task execution, Prometheus output, Grafana dashboard generation, memory delta, per-section timings, and operations per second.

`tsundere stress system` adds disposable project fixtures around that runtime check. It also tests compiler output, `.tsundere/runtime-build` emission, command discovery, package store harvest and hydration, Tsundere YAML workspace and lock snapshots, corrupt cache validation, store prune, platform runtime checks, updater release simulation, docs surface presence, per-section timings, and memory delta.

A `pass` status means the local stress checks delivered the expected counts. A `warn` status includes warnings such as missed IPC deliveries or task executions that timed out. Use `--json` when you want machine-readable results for a benchmark log or CI artifact.

Updater, Types, Plugins, And Generators

The CLI also exposes support commands. `tsundere version` prints the installed version and GitHub release repo. `tsundere updater check`, `self`, `info`, and `cron` manage release checks and self-update behavior. `tsundere types sync`, `inspect`, `clean`, and `doctor` manage Discord type metadata for editor support. `tsundere plugin install` handles plugin package names, GitHub repo shorthand, npm packages, local paths, and URLs. `tsundere generate` can scaffold API, route, service, command, model, and type files.

These commands are useful, but they are not all equally mature. Type metadata and plugin workflows are evolving. Generated files are simple starting points, not an application architecture. For production work, read the generated code, keep what helps, and replace what does not fit your project.

Runtime Config Example

{
  "name": "my-bot",
  "source": "src",
  "outDir": "build",
  "target": "typescript",
  "strict": true,
  "sourceMaps": true,
  "runtime": {
    "target": "node",
    "scale": "auto",
    "workers": "auto",
    "shards": "auto",
    "simulateShards": 1,
    "cache": {
      "backend": "memory"
    },
    "metrics": {
      "enabled": true,
      "port": 9100,
      "path": "/metrics",
      "format": "prometheus"
    },
    "tracing": {
      "enabled": false,
      "provider": "opentelemetry"
    }
  }
}

If `runtime` is a string, config loading treats it as `{ "target": "node" }` or another target string. If it is an object, nested `metrics`, `tracing`, and `cache` settings are merged across config layers. Metrics default to port `9100`, path `/metrics`, and Prometheus format.

Common CLI Mistakes

  • Running generated files directly and missing Tsundere's `.env` load, rebuild check, and runtime branding.
  • Using updater commands to update project dependencies. Use `tsundere update` for packages and `tsundere updater self --yes` for the CLI.
  • Expecting `reload` to perform production zero-downtime deployment today. It currently reports the staged runtime plan.
  • Expecting stress tests to contact Discord. The stress runner exercises the local distributed runtime foundation, not the real Discord gateway.
