Security

Security in Tsundere covers user projects, CLI behavior, package installation, release updates, installers, Discord permissions, and protected builds.

Overview

Tsundere runs in the JavaScript ecosystem, so secure defaults matter. The CLI should not delete user projects, installers should not hide risky changes, updater paths should be validated, logs should redact secrets, and generated Discord code should warn about missing permissions.

Why It Exists

A language ecosystem earns trust by being predictable. Developers should know what Tsundere writes, what it downloads, how it updates, where it stores caches, and how to report vulnerabilities.

When To Use These Checks

  • Before publishing a release bundle.
  • Before running installers or updater scripts as administrator.
  • Before deploying a bot with elevated Discord permissions.
  • Before enabling protected builds or embedding assets.

Secrets

DISCORD_TOKEN=replace-me
GITHUB_TOKEN=replace-me
DATABASE_URL=replace-me

Keep secrets in `.env` locally and deployment secret stores in production. Never commit real bot tokens, OAuth secrets, private keys, or webhook URLs.

Safe CLI Behavior

  • `tsundere install` should delegate dependency resolution to npm while validating Tsundere cache reuse.
  • `tsundere store prune` should only remove files inside the configured Tsundere store.
  • `tsundere cache clean` should only remove generated cache directories owned by Tsundere.
  • `tsundere updater` should verify release metadata before replacing global tooling.

Discord Permission Safety

guild.members.ban(user)

This operation needs BanMembers permission and can still fail if the target outranks the bot. Tsundere's permission intelligence roadmap warns about required permissions, hierarchy checks, and OAuth scope recommendations.

Updater Checklist

  • Fetch releases from the configured GitHub repository.
  • Prefer HTTPS release assets.
  • Show the current and target version before installing.
  • Surface security update notices when release notes contain security markers.
  • Keep package updates separate from self-updates.

Installer Safety

The Windows installer should verify existing installs, create an uninstaller, update PATH intentionally, and never remove user-created projects during uninstall. It should display what it is doing with a progress bar instead of hiding work in console windows.

Common Mistakes

  • Do not run arbitrary GitHub plugin links without reviewing the source.
  • Do not grant Administrator permission to bots that only need a few permissions.
  • Do not store warning databases or logs in public directories when they include user moderation data.
  • Do not publish source maps for protected production builds.

Performance Notes

Security checks should be cheap during normal builds and deeper during `tsundere doctor`, CI, or release workflows. Expensive scans belong in explicit commands or GitHub Actions so day-to-day development stays fast.
