Versions

This page explains how Tsundere knows its own version, how it checks GitHub releases, and why updating the CLI is separate from updating project dependencies. The behavior here comes from `src/cli.ts`, `src/updater.ts`, and the `tsundere` metadata in `package.json`.

The short version: `tsundere version` reads the installed package version, `tsundere updater` checks a GitHub release feed, `tsundere updater self --yes` can install a release tarball globally with npm, and `tsundere update ` updates dependencies inside your project.

What This Page Teaches

  • What `tsundere version` prints and where those values come from.
  • How Tsundere chooses the GitHub repository used for release checks.
  • How `check`, `info`, `self`, and `cron` updater actions behave.
  • Which release assets the self-updater accepts.
  • How security update notices are cached.
  • How CLI updates differ from npm package updates in a project.

Check the Installed CLI Version

The `version` command reads `package.json` next to the built CLI. If the package file cannot be found, the helper falls back to `0.0.0`, which is a sign that the install or build layout is wrong.

tsundere version

Expected output shape:

Tsundere 0.1.1
GitHub releases: TsundereLang/tsundere

That second line is important. It tells you which repository the updater will query unless you override it for testing.

Where the Release Repository Comes From

The CLI resolves the update repository in this order:

  1. `TSUNDERE_UPDATE_REPO`, when the environment variable is set.
  2. `package.json` metadata at `tsundere.githubRepo`.
  3. The hardcoded fallback `TsundereLang/tsundere`.

The repo format is `owner/repo`, because `src/updater.ts` calls the GitHub API endpoint for the latest release.

PowerShell

$env:TSUNDERE_UPDATE_REPO = "TsundereLang/tsundere"
tsundere updater check

Linux or macOS Shell

TSUNDERE_UPDATE_REPO="TsundereLang/tsundere" tsundere updater check

Updater Commands

`tsundere updater` defaults to `check`. The allowed actions are `check`, `info`, `self`, and `cron`. Anything else prints the usage error from `src/cli.ts`.

CommandReal Behavior
tsundere updaterSame as `tsundere updater check`.
tsundere updater checkPrints current version, release repo, latest release, update status, and the self-update hint.
tsundere updater infoUses the same latest-release path as `check`; useful when you want release visibility without installing.
tsundere updater self --dry-runSelects the release asset and reports what would be installed without changing the machine.
tsundere updater self --yesDownloads the selected CLI tarball and runs `npm install -g` on it.
tsundere updater self --force --yesAllows reinstalling even when the latest release is not newer than the current version.
tsundere updater cron installInstalls an automatic daily update check using Windows Task Scheduler or a Unix crontab entry.
tsundere updater cron statusChecks whether the scheduled updater exists.
tsundere updater cron removeRemoves the scheduled updater entry.

What the GitHub Check Reads

The latest-release function calls GitHub's `releases/latest` API with an Accept header for GitHub JSON and a `User-Agent` like `tsundere-cli/0.1.1`. It reads the release tag, normalizes a leading `v`, keeps the release URL, keeps the assets array, and preserves the release body for security-notice checks.

https://api.github.com/repos/TsundereLang/tsundere/releases/latest

If GitHub cannot be reached, returns a non-OK response, or does not include a valid tag, `tsundere updater check` prints:

Could not read the latest release. Check your network connection or TSUNDERE_UPDATE_REPO.

Release Asset Rules

The self-updater does not install arbitrary release files. It looks for a Tsundere CLI tarball in this order:

  1. A versioned tarball matching `tsundere-cli-1.2.3...tgz`.
  2. An exact `tsundere-cli.tgz` asset.
  3. Any tarball name starting with `tsundere-cli` and ending in `.tgz`.

If the latest release does not include a matching tarball, `selfUpdate` returns code `1` with the message `Latest release is missing a tsundere-cli tarball asset.` This usually means the release packaging step failed or the release was drafted without upload assets.

Version Comparison

Tsundere compares release versions by removing a leading `v`, splitting on dots, parsing numeric parts, and comparing each position. That means `v0.1.2` and `0.1.2` compare the same. It is a simple numeric comparison, not a full semantic-version parser with prerelease ordering.

ComparisonResult
0.1.2 vs 0.1.1Newer release available.
v0.1.1 vs 0.1.1Same version.
0.1.1 vs 0.1.2Installed version is behind the target.

Automatic Checks and Security Notices

The CLI can show a security update notice before running higher-risk or common commands: `dev`, `start`, `build`, `install`, `add`, and `update`. This happens in `maybeWarnSecurityUpdate` before command handling continues.

The notice path checks the latest release, compares it to the installed version, then looks for words such as `security`, `critical`, `vulnerability`, `cve-`, `exploit`, `hotfix`, or `patch now` in the tag, version, or release body. Results are cached for 24 hours in:

~/.tsundere/security-update-cache.json

Set `TSUNDERE_NO_UPDATE_NOTICE=1` when you want command output to stay quiet in automation.

Scheduled Update Checks

`tsundere updater cron` only schedules a daily check. It does not silently install new versions. On Windows, Tsundere uses `schtasks` with the task name `Tsundere Daily Update Check`. On Linux and macOS, it edits crontab entries marked with `# tsundere-daily-update-check`.

Install or Inspect

tsundere updater cron install
tsundere updater cron status
tsundere updater cron remove

Custom Time

tsundere updater cron install --time 09:30

The Unix implementation currently writes a `0 10 * * *` line regardless of the `--time` value passed into the CLI. The CLI accepts the option, but the Unix cron helper does not yet apply it. Windows passes the selected time to `schtasks`.

Project Package Versions Are Separate

Do not use `tsundere updater self` to update `discord.js`, `zod`, `mysql2`, or any other dependency in your bot project. The self-updater changes the globally installed Tsundere CLI. Project packages go through the package optimizer, which wraps npm and preserves `package.json` plus `package-lock.json` behavior.

tsundere install discord.js
tsundere add zod
tsundere update discord.js
tsundere remove zod

After a successful install/update/remove, the package optimizer can also refresh `tsundere-workspace.yaml` and `tsundere-lock.yaml`, but npm remains the source of package resolution through `package-lock.json`.

Troubleshooting

`tsundere version` says `0.0.0`

The CLI could not find or read its package file. Rebuild from source with `npm run build`, check that `dist/cli.js` is being run from the expected package root, or reinstall the CLI release.

The latest release cannot be read

Check network access, GitHub availability, and `TSUNDERE_UPDATE_REPO`. The repo value must be `owner/repo`, not a full URL.

A release exists but self-update fails

Open the release assets and confirm there is a `.tgz` file named like `tsundere-cli-0.1.1.tgz` or `tsundere-cli.tgz`. Without that asset, the updater refuses to install.

Scheduled updates do not appear on Linux

Run `crontab -l` and look for `# tsundere-daily-update-check`. If `crontab` is missing on the machine, install or enable the system's cron service, then rerun `tsundere updater cron install`.
