Transition to Yuri

This page is for JavaScript, TypeScript, and Python bot developers who want to move into Tsundere without guessing what Yuri really is. Yuri is not a finished independent language runtime. In this repo, `.yuri` files are compiled by `src/compiler/transpile.ts` through targeted source transforms, then the project builder emits runnable JavaScript into `.tsundere/runtime-build`.

The safest way to learn Yuri today is to keep code close to JavaScript and TypeScript, then use the few Tsundere conveniences that are actually implemented: `env`, `log`, `print`, parenthesis-light `if`, `fn`, Discord import rewriting, native Discord blocks, Spanglish aliases, source maps, strict diagnostics, command discovery, and the bundled `@tsundere/discord` wrapper.

What This Page Teaches

  • How `.yuri` maps to JavaScript or TypeScript in the current compiler.
  • Which shorthand forms are implemented in `src/compiler/transpile.ts`.
  • How to port Discord.js event handlers and slash commands without losing control of the code.
  • How Python bot habits translate into Tsundere's Node and Discord.js-shaped model.
  • Which ideas are experimental and should not be treated like stable language features yet.

The Mental Model

A `.yuri` file is source code that Tsundere reads, lightly normalizes, and writes as `.ts` or `.js` depending on `target` in `tsundere.config.json`. The project builder in `src/compiler/project.ts` then writes TypeScript-style output into the configured `outDir`, transpiles runtime JavaScript when needed, refreshes `.tsundere/runtime/discord`, and emits runnable files under `.tsundere/runtime-build`.

LayerWhat Happens
src/*.yuriYour source files. Keep these close to JS/TS while using supported Yuri shorthand.
src/compiler/transpile.tsNormalizes implemented syntax and emits diagnostics. This is not a full AST parser yet.
build/Intermediate `.ts` or `.js` output based on project config.
.tsundere/runtime-build/Runnable Node JavaScript. `tsundere start` runs `.tsundere/runtime-build/main.js`.
.tsundere/runtime/discordLocal file package for the bundled Discord wrapper used by generated projects.

That model matters because it keeps expectations honest. If normal JavaScript or TypeScript syntax is valid after Tsundere's transforms, it is usually the least surprising path. If a proposed Yuri feature is not listed here or used in `docs/examples`, test it with `tsundere build` before relying on it.

From JavaScript or TypeScript

Most Discord.js code moves over by keeping the event shape and using Tsundere's wrapper helpers where they make the code clearer. Imports from `tsundere/discord` are rewritten to `@tsundere/discord`, and imports that already use `@tsundere/discord` are preserved.

Discord.js Style

client.on("interactionCreate", async (interaction) => {
  if (!interaction.isChatInputCommand()) return
  if (interaction.commandName === "ping") {
    await interaction.reply("pong")
  }
})

Yuri Style

client.on("interactionCreate", async (interaction) => {
  if interaction.isCommand("ping") {
    await interaction.reply("pong")
  }
})

The `if interaction.isCommand("ping") {` form is one of the currently implemented parenthesis-light transforms. It becomes `if (interaction.isCommand("ping")) {` before the runtime is emitted. This is useful, but keep complex conditions simple. If a condition starts getting hard to read, use normal JavaScript parentheses.

Runtime Prelude Helpers

The compiler adds small prelude bindings only when it sees the matching names. If your code uses `env.`, Tsundere emits `const env = process.env`. If your code uses `log(`, it emits a bound `console.log`. If your code uses `print(`, it emits another bound `console.log` alias.

const client = new Client({
  token: env.DISCORD_TOKEN,
  intents: [Intents.Guilds]
})

log("starting bot")
print("same console output path")

This is why examples can use `env.DISCORD_TOKEN` without an import. It is still Node environment access. Put secrets in `.env` for local development, because `src/compiler/project.ts` merges `.env` values into the child process started by `tsundere dev` and `tsundere start`.

Syntax That Actually Lowers Today

The supported shorthand is deliberately small. These forms are implemented as source transforms in `src/compiler/transpile.ts`, so whitespace and shape can matter more than it would in a mature parser.

Yuri FormEmitted Shape
fn name(args) {}`function name(args) {}`
async fn name(args) {}`async function name(args) {}`
if value {}`if (value) {}`
catch error {}`catch (error) {}`
client bot { token ... intents [...] }`const bot = new Client({ token, intents })`
on ready() {}`bot.on("ready", async () => {})`
embed { title "Hi" }`Embed.create().title("Hi")`
await interaction.reply { content "pong" }`await interaction.reply({ content: "pong" })`
match(value) {}Warns with `YURI210` and emits a runtime `match(value)({` shape.

There is also a Spanglish layer: `usar discord { Client }`, `usar pkg "name"`, `verdad`, `falso`, `nulo`, `ahora()`, `Text`, `Num`, `Bool`, `Void`, `tipo`, `sea`, `retorna`, `espera`, `lanza`, `intenta`, `captura`, `si`, `sino`, `para`, and `mientras`. These are real transforms, but teams that want the clearest migration path usually start with JavaScript-like Yuri first.

Discord Diagnostics You Will Notice

When `strict` is enabled in config, Tsundere adds warnings and errors that are Discord-specific. These do not replace TypeScript or Discord.js validation. They are source checks that catch common bot mistakes early.

  • `YURI101` warns when strict code uses `any`.
  • `DISCORD010` errors when a custom ID is longer than Discord's 100 character limit.
  • `DISCORD005` errors when embed title or description builder strings exceed Discord limits.
  • `DISCORD002` warns when event handlers appear to need intents such as `GuildMessages`, `GuildMembers`, `GuildPresences`, or `GuildVoiceStates`.
  • `DISCORD003` warns when message content is read without an obvious `MessageContent` intent.
  • `DISCORD020` warns around operations such as ban, kick, timeout, channel creation, role changes, and audit log reads.

These diagnostics are intentionally practical. If a warning appears, read the hint, then check the Discord developer portal and your bot invite permissions. Some intents are privileged and cannot be fixed by code alone.

From Python Bot Frameworks

Python bot frameworks often feel decorator-driven: define an event or command, and the framework wires it up. Tsundere is closer to Discord.js. You create a client, attach event handlers, define commands, and run the bot through Node. Command discovery can reduce loader boilerplate, but it does not hide the generated command metadata from you.

Python Habit

@bot.event
async def on_ready():
    print("online")

@bot.command()
async def ping(ctx):
    await ctx.reply("pong")

Tsundere Habit

client.once("ready", () => {
  log("online")
})

export const command = Slash.command("ping")
  .description("Reply with pong")

A command file like `src/commands/ping.yuri` can be discovered by `tsundere commands sync` or by `tsundere dev` before the runtime starts. The output goes to `.yuri-cache/discord.commands.json`, so you can inspect what Tsundere found.

A Safe Migration Path

  1. Create or open a Tsundere project and keep `target` as `typescript` while learning.
  2. Move one Discord event handler at a time into `src/main.yuri` or a small imported module.
  3. Replace `process.env.X` with `env.X` only when it makes the code clearer.
  4. Use normal `client.on`, `client.once`, `Slash.command`, `Embed.create`, and interaction reply calls first.
  5. Move slash command definitions into `src/commands` after the bot starts cleanly.
  6. Run `tsundere build` after each syntax change and inspect `build/` or `.tsundere/runtime-build/` if behavior surprises you.
  7. Run `tsundere dev` only after the project builds, because dev mode will rebuild and restart on changes.
tsundere build
tsundere commands sync
tsundere dev
tsundere start

Common Mistakes

Treating Yuri Like a Full Parser

The current compiler uses targeted regular-expression and block transforms. Deeply nested experimental syntax may not lower the way you expect. When in doubt, write ordinary JS/TS-shaped code.

Running Generated Files By Hand

Use `tsundere start` instead of manually running `.tsundere/runtime-build/main.js`. The CLI builds if needed, loads `.env`, and rewrites runtime error footers to Tsundere Runtime branding.

Forgetting Intent Warnings

If you handle messages, members, presences, or voice state, configure the matching intent in code and in Discord's developer portal when the intent is privileged.

Assuming Command Discovery Registers Commands

Discovery writes local metadata. Your bot code or command sync workflow still needs to register commands through the Discord API path used by your project.

Troubleshooting

The build output looks weird

Open the matching file in `build/` first. If `target` is `typescript`, then check `.tsundere/runtime-build/` after TypeScript transpilation. This usually shows whether the surprise came from Yuri lowering or from TypeScript-to-JavaScript emission.

A Discord helper cannot be imported

Run `tsundere runtime install` and then `tsundere install`. Generated projects depend on `@tsundere/discord` through `file:.tsundere/runtime/discord`, so the local runtime package has to exist.

A command file is ignored

Check `commands.directory` in `tsundere.config.json`. The default command discovery path is `./src/commands`, and route-based discovery uses the file path to build command routes.
