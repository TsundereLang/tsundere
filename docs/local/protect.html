Tsundere Protect

Tsundere Protect is an early protection pass for the JavaScript emitted into `.tsundere/runtime-build`. It runs after `.yuri` files have been compiled and after TypeScript output has been converted to runtime JavaScript. The implementation lives in `src/compiler/protect.ts`, and the CLI enables it with `tsundere build --protect`.

Protect is useful for adding build fingerprints, removing obvious source-map comments, encoding static strings, and making casual copying harder. It is not encryption for secrets, not licensing enforcement, not DRM, and not a guarantee that JavaScript cannot be reverse engineered. Treat it as traceable friction, not a security boundary.

The Real Build Flow

When you run `tsundere build --protect standard`, the CLI parses `--protect` in `src/cli.ts`, passes the selected profile and optional seed into `buildProject`, and `src/compiler/project.ts` applies `protectJavaScript` to each emitted runtime file. Protection happens only when `--protect` is present.

  1. Tsundere loads `tsundere.config.json`.
  2. It scans the configured source directory for `.yuri` files.
  3. It compiles `.yuri` files into `build/` or the configured `outDir`.
  4. It emits runnable JavaScript into `.tsundere/runtime-build/`.
  5. If `--protect` was passed, it transforms the runtime JavaScript before writing it.
  6. It writes `.tsundere/runtime-build/tsundere-protect.json` with profile, seed, file names, and build IDs.
tsundere build --protect standard
tsundere build --protect advanced
tsundere build --protect maximum
tsundere build --protect advanced --seed release-2026-06-06

If `--protect` is passed without a valid profile name, the CLI falls back to `standard`. Valid profiles are exactly `standard`, `advanced`, and `maximum`.

Profile Behavior

ProfileWhat It Adds TodayWhat It Does Not Add
standardComment stripping, source-map comment stripping, static string encoding, build ID insertion, whitespace minification, metadata.No guard function, no integrity token, no wrapper layer.
advancedEverything in `standard`, plus generated guard code and a small integrity token check.No control-flow flattening, no runtime virtualization, no real tamper-proofing.
maximumEverything in `advanced`, plus an extra generated wrapper gate.No layered decryptor system, no secret protection, no licensing server.

The old roadmap language talks about stronger layers such as control-flow flattening, asset protection, environment protection, anti-debugging, and licensing. Those are future ideas. The current source implements the smaller set above.

What String Encoding Means

Protect leaves import lines alone, then replaces ordinary quoted strings with calls to a generated decoder helper. The encoded payload is base64 text produced after XORing characters with a build key. Template strings that contain interpolation are left alone for compatibility.

This is obfuscation, not cryptography. Anyone with the protected file has the decoder helper and the encoded strings. Do not place Discord tokens, database passwords, API keys, license secrets, or customer secrets in source code and expect Protect to hide them. Use `.env`, deployment secrets, or server-side storage.

Build IDs and Seeds

Each protected file receives a build ID shaped like `7A4F-91D3-C2E8`. The ID is generated from the profile, entropy, and sometimes the current time. When you pass a seed that is not `auto`, the fingerprint uses that seed and omits the time component, so repeated builds with the same profile and seed can produce the same build ID.

tsundere build --protect standard --seed customer-a
tsundere build --protect standard --seed customer-b
tsundere build --protect standard --seed auto

The metadata records the seed as the provided value, or `auto` when no seed is supplied. Use named seeds when you need repeatable fingerprints for an internal build process. Use no seed or `auto` when you want each build to vary.

Metadata File

Protected builds write one metadata file in the runtime output directory:

.tsundere/runtime-build/tsundere-protect.json

The file contains `generatedAt`, `profile`, `seed`, and a `files` list. Each file entry includes the emitted runtime file path, the build ID, and the profile used for that file.

{
  "generatedAt": "2026-06-06T12:00:00.000Z",
  "profile": "advanced",
  "seed": "release-2026-06-06",
  "files": [
    {
      "file": "main.js",
      "buildId": "7A4F-91D3-C2E8",
      "profile": "advanced"
    }
  ]
}

Fingerprint Inspection

`tsundere fingerprint inspect` defaults to `.tsundere/runtime-build`. The CLI keeps the target inside the current project root through the shared path-safety helper before reading anything.

tsundere fingerprint inspect
tsundere fingerprint inspect .tsundere/runtime-build
tsundere fingerprint inspect .tsundere/runtime-build/main.js

If the target is a directory and contains `tsundere-protect.json`, the report prints the profile and every protected file. If the target is a directory without metadata, it prints `No Tsundere fingerprint metadata found in this directory.` and returns exit code `1`.

If the target is a file, the CLI scans for build-ID-shaped values with the pattern `0000-0000-0000` using uppercase hex characters. If it finds IDs, it prints each unique build. If not, it prints `No Tsundere fingerprint detected.` and returns exit code `1`.

Tsundere Fingerprint Report
  profile: advanced
  main.js: 7A4F-91D3-C2E8 (advanced)

What To Check After a Protected Build

  1. Run `tsundere build --protect standard` and make sure the command exits successfully.
  2. Open `.tsundere/runtime-build/tsundere-protect.json` and confirm every expected runtime file is listed.
  3. Run `tsundere fingerprint inspect` and confirm it reports the same build IDs.
  4. Run `tsundere start` in a safe test environment and confirm the protected runtime still starts.
  5. If using Discord, test interactions with a test bot before shipping a protected build.

Protect transforms JavaScript text. Most simple projects should keep working, but generated output can expose edge cases in string handling or minification. Test the protected output, not just the unprotected dev build.

Windows and Linux Notes

Protect itself uses Node APIs and writes into the project directory, so the same command shape works on Windows, macOS, and Linux. The practical differences are shell syntax and path display.

Windows PowerShell

tsundere build --protect advanced --seed release-test
tsundere fingerprint inspect .tsundere/runtime-build

Linux or macOS Shell

tsundere build --protect advanced --seed release-test
tsundere fingerprint inspect .tsundere/runtime-build

On Linux, make sure the CLI is executable if you installed from a source checkout. `tsundere doctor` checks Linux executable permissions for bundled scripts and reports Node and npm availability.

Common Mistakes

Putting Secrets in Source

Protect does not make source-embedded secrets safe. Keep secrets in environment variables or infrastructure secret stores.

Expecting Dev Mode Protection

`tsundere dev` does not pass protect options. Use `tsundere build --protect ...` when you want protected runtime output.

Inspecting the Wrong Directory

The metadata file lives in `.tsundere/runtime-build`, not `build/`. `build/` is the intermediate output directory.

Assuming Maximum Means Complete

`maximum` currently adds a small wrapper gate on top of advanced behavior. It is not a full virtualization or commercial licensing system.

Troubleshooting

`fingerprint inspect` says no metadata was found

Run a protected build first. A normal `tsundere build` removes and recreates `.tsundere/runtime-build` without writing `tsundere-protect.json`.

The protected bot fails but the dev bot works

Inspect the emitted `.tsundere/runtime-build/main.js`, try the lower `standard` profile, and check whether unusual strings or generated code were affected by minification. Keep a small reproduction file if you open an issue.

Build IDs do not change

Check whether you passed a fixed `--seed`. Fixed seeds are meant to make fingerprints repeatable. Remove the seed or use `--seed auto` for varied IDs.

A path outside the project is rejected

That is expected. Fingerprint inspection uses project-bounded path resolution so a malformed path cannot read arbitrary files outside the current project.
