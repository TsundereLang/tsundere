Discord Layouts

Tsundere does not require one folder structure, but the CLI template and command discovery are built around a small `src/main.yuri` file and discoverable command files under `src/commands`.

Recommended Shape

src/
  main.yuri
  commands/
    ping.yuri
    admin/
      ban.yuri
  events/
    ready.yuri
    interactionCreate.yuri
  components/
    banButton.yuri
  services/
    members.yuri

Only `src/main.yuri` and `src/commands/ping.yuri` are created by the Discord starter today. The other folders are a recommended structure for real bots as they grow. Start small, then split events, components, and services when the main file becomes hard to scan.

Main File

Keep `main.yuri` responsible for creating the client, wiring handlers, and logging in. The compiler reads from `src` by default and emits runtime output into `.tsundere/runtime-build`.

import { Client, Intents } from "@tsundere/discord"
import { registerReady } from "./events/ready"
import { registerInteractions } from "./events/interactionCreate"

const client = new Client({
  token: env.DISCORD_TOKEN,
  intents: [
    Intents.Guilds,
    Intents.GuildMessages,
    Intents.MessageContent
  ]
})

registerReady(client)
registerInteractions(client)

await client.login()

The compiler preserves normal imports and exports, then adds `.js` extensions during runtime emission. That means `./events/ready` in source can become `./events/ready.js` in `.tsundere/runtime-build`. Keep imports relative to source files and let the runtime emitter handle Node's ESM extension requirements.

Command Files

Command discovery scans `.yuri` files in the configured command directory. It reads `Slash.command("name")` and `.description("text")`, then writes `.yuri-cache/discord.commands.json` when you run `tsundere dev` or `tsundere commands sync`.

import { Slash } from "@tsundere/discord"

export default Slash.command("ping")
  .description("Check whether the bot is online")

Discovery reads source text. It does not execute modules, import your command files, or evaluate variables. Literal builder calls are the easiest path: `Slash.command("ping")` and `.description("...")`. Dynamic command names may work at runtime if you import and register them yourself, but the manifest generator will not infer them.

The generated manifest records `generatedAt`, the resolved command directory, and each command's file, name, description, group, route, and routeBased flag. Use it as a local metadata file for tooling, not as a replacement for Discord command registration.

Route-Based Commands

With `routeBased: true`, a file path such as `src/commands/admin/ban.yuri` becomes the route `admin ban`. With `routeBased: false`, the command name comes from the command metadata or the file name.

{
  "commands": {
    "discovery": true,
    "routeBased": true,
    "directory": "./src/commands",
    "groups": {
      "./src/commands/admin": {
        "routeBased": true,
        "groupName": "admin"
      }
    }
  }
}

Group overrides are matched by checking whether a command file path starts inside the configured group folder. Path safety helpers resolve those folders inside the project before walking files. If a group is not being applied, check the folder path in `tsundere.config.json` and inspect `.yuri-cache/discord.commands.json` after running `tsundere commands sync`.

Router Style

The runtime exports `InteractionRouter` and a shared `router`. A router can bind to the client and route command names, typed components, modals, and selects from one `interactionCreate` listener.

import { Component, router } from "@tsundere/discord"

const BanButton = Component.define<{ userId: string }>("ban")

router
  .command("ping", async (ctx) => {
    await ctx.reply({ content: "pong" })
  })
  .button(BanButton, async (ctx) => {
    await ctx.reply({ content: `Ban ${ctx.data.userId}` })
  })
  .bind(client)

`InteractionRouter` is useful when you want one central interaction listener. It routes by `interaction.commandName` for slash commands and by parsed custom ID namespace for buttons, modals, and selects. The shared `router` export is a convenience instance; larger projects may prefer creating their own router instances for tests or separate modules.

For typed buttons, route data comes from the encoded custom ID. For modals and selects, the current runtime has basic routing hooks, but modal field parsing is still minimal. Test those flows carefully and keep the data model simple until the typed modal/select APIs mature.

When To Split Files

Split files when a command, event, or service has its own state or tests. Keep shared Discord logic in services and export small registration functions from event files. The compiler preserves normal import/export syntax, so source files can reference each other with standard relative imports.

Good split points are command handlers, interaction routers, role utilities, configuration loaders, and API clients. Avoid splitting every tiny helper on day one; too many files can make the beginner loop harder. The goal is to keep `main.yuri` boring: create the client, bind handlers, and login.

Layout Troubleshooting

  • If imports fail at runtime, check the source import path and confirm the generated `.js` file exists under `.tsundere/runtime-build`.
  • If command discovery misses a command, keep the command name and description literal or inspect the configured command directory.
  • If a route group looks wrong, run `tsundere commands sync` and read `.yuri-cache/discord.commands.json`.
  • If the main file grows too large, move business logic into services before moving basic client startup code.
