Discord Guide

Tsundere ships a local `@tsundere/discord` bridge that wraps Discord.js with a smaller API surface for `.yuri` projects. It is still Discord.js underneath when you use the real gateway, but your source imports the Tsundere helpers and the compiler emits runnable JavaScript into `.tsundere/runtime-build`.

What This Layer Provides

The runtime exports `Client`, `Intents`, `Partials`, builders such as `Slash`, `Embed`, `Button`, `SelectMenu`, `Modal`, `TextInput`, `Row`, Components v2 helpers, interaction helpers, a typed interaction router, cache helpers, REST helpers, prefix parsing, and runtime member role helpers.

It does not replace Discord.js. It keeps the bot code compact while preserving normal Discord token, intent, gateway, REST, and interaction behavior.

When `gateway` is not set to `"mock"`, the client creates a real Discord.js gateway client. When `gateway: "mock"` is used, login emits `ready` without network access, which is how the unit tests exercise runtime behavior safely.

Client Setup

Create a client with a token and numeric intent values from `Intents`. The real gateway is used by default. Tests and local simulations can pass `gateway: "mock"`.

import { Client, Intents } from "@tsundere/discord"

const client = new Client({
  token: env.DISCORD_TOKEN,
  intents: [
    Intents.Guilds,
    Intents.GuildMessages,
    Intents.MessageContent
  ]
})

client.once("ready", () => {
  log(`Online as ${client.user.tag}`)
})

await client.login()

The `Client` constructor stores your intents, partials, shard setting, REST client, cache manager, guild manager, and channel manager. If a token is passed in the constructor, `login()` can be called without arguments. If not, pass the token to `login(token)`. Missing tokens throw `Missing Discord token. Pass token to Client or login(token).`

Intents are numeric values from the Tsundere `Intents` enum. They are forwarded to Discord.js. Tsundere can warn about missing intents in strict diagnostics, but it cannot enable privileged intents in the Discord Developer Portal for you.

Interactions

`interactionCreate` receives a normalized interaction with narrowing helpers and reply methods. Options can be read from `interaction.options` with helpers such as `string`, `number`, `user`, `channel`, and `role` when Discord provides them.

client.on("interactionCreate", async (interaction) => {
  if (interaction.isCommand("ping")) {
    await interaction.reply({ content: "pong" })
  }

  if (interaction.isButton("ping:again")) {
    await interaction.reply({ content: "still pong" })
  }
})

The normalized interaction includes `id`, `token`, optional `guildId`, `channelId`, `user`, `member`, `guild`, `channel`, `commandName`, `customId`, `options`, and methods for `reply`, `deferReply`, `editReply`, `followUp`, and `deleteReply`. Reply payloads can contain content, embeds, components, attachments, and the current `ephemeral` option.

The wrapper converts Tsundere components into Discord API component payloads when replies are sent. Buttons, select menus, and text inputs are normalized into Discord component types. Components v2 helpers are currently represented as structured payload objects and should be tested against Discord behavior before relying on them in a production bot.

Slash Commands

The `Slash` builder stores command JSON and can register directly through the client REST helper. If the application ID is not known yet, registration waits for the `ready` event.

import { Slash } from "@tsundere/discord"

export default Slash.command("avatar")
  .description("Show a user's avatar")
  .option("user", "target", "User to inspect", { required: false })
  .guildOnly()

`Slash.option` takes the option type first, then the option name, then the description. Supported option types are `string`, `integer`, `number`, `boolean`, `user`, `channel`, `role`, and `mentionable`. Subcommands and groups are available through `.subcommand(...)` and `.group(...)`.

For registration, `Slash.register(client, scope)` calls `registerSlashCommands` with a single command. `registerSlashCommands` writes to either the global command route or a guild route when `scope.guildId` is provided. If the client user is still the runtime placeholder, registration waits for `ready` so it can use the application ID.

Embeds And Components

Builders expose `toJSON()` so replies can pass Tsundere objects directly. `Embed.validate()` checks common Discord embed limits before you send the payload.

import { Button, Embed, Row } from "@tsundere/discord"

const embed = Embed.create()
  .title("Status")
  .description("The bot is online")
  .color("#ff7ab6")
  .timestamp()

const row = Row.of(
  Button.secondary("ping:again").label("Ping again")
)

await interaction.reply({
  embeds: [embed],
  components: [row]
})

`Embed.validate()` checks title, description, field count, and total text length against Discord embed limits. It returns `{ ok, errors }`; it does not send anything. Use it before replying when user-provided data can make an embed too large.

`Row.of(...)` accepts builders with `toJSON()` and creates a row payload. Use rows for buttons, select menus, and text inputs. A link button uses `Button.link(url)` and does not need a custom ID. Non-link buttons need custom IDs because Discord sends those IDs back in interactions.

Typed Custom IDs

`Component.define` serializes small JSON data into a Discord custom ID and enforces the 100 character Discord limit. It returns a button builder, so change style and label on that returned button.

import { Component } from "@tsundere/discord"

const BanButton = Component.define<{
  userId: string
}>("ban")

const button = BanButton
  .create({ userId: member.id })
  .style("danger")
  .label("Ban")

const data = BanButton.parse(interaction.customId)

The serialized ID format is `namespace:base64url-json`. That keeps handler code cleaner, but the data still has to fit inside Discord's 100 character custom ID limit after encoding. Store large state in your database and put only a short ID in the custom ID.

Runtime Managers

The client exposes `guilds.fetch`, `channels.fetch`, `guild.members.fetch`, `member.roles.add`, `member.roles.remove`, `member.kick`, and `member.timeout`. In real Discord mode these call Discord.js. In mock/runtime mode they use local runtime objects so tests can run without the network.

Mapped gateway events provide plain Tsundere objects rather than raw Discord.js classes. That is why `member.roles.add(roleId)` exists as a method on the normalized member roles object. If you bypass the wrapper and pass raw objects into helper code, check whether the object uses the Tsundere shape or the Discord.js shape before calling methods.

Troubleshooting Discord Runtime Code

  • If login fails immediately, confirm `DISCORD_TOKEN` is present and the token belongs to a bot application.
  • If message content is empty, add `Intents.MessageContent` in code and enable the privileged intent in the Developer Portal when required.
  • If member join events do not fire, add `Intents.GuildMembers` and enable the Server Members intent when Discord requires it.
  • If role helpers fail, confirm the bot has Manage Roles and that its highest role is above the target role.
  • If `@tsundere/discord` cannot resolve, repair the local runtime with `tsundere runtime install` and reinstall dependencies.
