Discord Intelligence

Discord Intelligence is Tsundere's name for the Discord-aware checks and editor metadata that exist around `.yuri` code. In the current repo, this is not one finished dashboard. It is three working foundations: compiler diagnostics in `src/compiler/transpile.ts`, VS Code completions and hovers in `packages/vscode-tsundere/src/discordIntellisense.ts`, and type metadata generation in `src/type-bridge`.

The important expectation: Tsundere can catch some common Discord mistakes early, but it does not replace Discord's developer portal, bot permissions, runtime testing, or Discord.js documentation. Use it as a helpful warning layer, not as proof that a bot is deploy-ready.

What This Page Teaches

  • Which Discord diagnostics are implemented today.
  • Why strict mode controls most compiler intelligence.
  • What the VS Code extension suggests and when suggestions appear.
  • What `tsundere types sync`, `inspect`, `doctor`, and `clean` write or read.
  • Which roadmap ideas, such as dashboards and full compatibility checks, are not built yet.
  • How to troubleshoot missing or noisy Discord intelligence.

The Three Current Layers

LayerSourceCurrent Behavior
Compiler diagnostics`src/compiler/transpile.ts`Warns or errors on selected Discord limits, intents, message content, and permission-sensitive operations when `strict` is enabled.
Editor intelligence`packages/vscode-tsundere/src/discordIntellisense.ts`Adds completions, hovers, and signature help for common Discord events, interactions, builders, components, modals, selects, router helpers, and auto imports.
Type bridge`src/type-bridge`Extracts installed Discord package declarations and writes `.yuri-cache` JSON metadata for events, builders, imports, docs, types, and cache keys.

Compiler Diagnostics

The compiler adds Discord diagnostics only when config has `strict: true`. That is the default in generated projects. The diagnostics are source-pattern checks, not a full semantic analyzer, so they are intentionally narrow. They are still useful because they catch mistakes that often cause Discord bots to fail after deployment.

CodeWhen It AppearsSeverity
DISCORD010A `customId` string is longer than Discord's 100 character limit.error
DISCORD005An embed `.title("...")` exceeds 256 characters or `.description("...")` exceeds 4096 characters.error
DISCORD002An event such as `messageCreate`, `guildMemberAdd`, `presenceUpdate`, or `voiceStateUpdate` appears without the matching intent name in source.warning
DISCORD003A message event reads `message.content` or `msg.content` without an obvious `MessageContent` intent.warning
DISCORD020Source appears to call ban, timeout, kick, channel creation, role operations, or audit log reads.warning

The same strict path also warns with `YURI101` when code uses `any`. That warning is not Discord-specific, but it matters for Discord bots because untyped interaction handlers are where many runtime mistakes hide.

Intent Warnings

Intent warnings are simple and practical. The compiler searches for `.on("eventName"` patterns and checks whether the required intent text appears somewhere in the source. It does not parse the final client options object semantically, and it does not check the Discord developer portal.

import { Client, Intents } from "@tsundere/discord"

const client = new Client({
  token: env.DISCORD_TOKEN,
  intents: [Intents.Guilds]
})

client.on("messageCreate", async (message) => {
  log(message.content)
})

With strict diagnostics enabled, this shape can warn that `messageCreate` needs `GuildMessages` and that reading message content may need `MessageContent`. You still need to enable privileged intents in Discord's developer portal when Discord requires that outside your code.

Permission Warnings

Permission diagnostics look for method patterns that commonly need bot permissions or role hierarchy checks. They do not inspect your server roles, OAuth invite URL, or runtime member hierarchy.

Warned Operations

  • `ban(...)` needs `BanMembers`.
  • `timeout(...)` needs `ModerateMembers`.
  • `kick(...)` needs `KickMembers`.
  • `channels.create(...)` or `createChannel(...)` needs `ManageChannels`.
  • `roles.create(...)`, `setRole`, or `addRole(...)` needs `ManageRoles`.
  • `fetchAuditLogs(...)` needs `ViewAuditLog`.

What to Check Manually

  • The bot invite includes the required permission.
  • The bot role is above the target role or member.
  • The invoking user has the permission your command expects.
  • The server has enabled required privileged intents.
  • Your error handling covers Discord `Missing Permissions` and `Missing Access` responses.

Embed and Custom ID Limits

The compiler currently catches two common hard limits directly in source strings: custom IDs longer than 100 characters, embed titles longer than 256 characters, and embed descriptions longer than 4096 characters.

Button.primary("really-long-custom-id...")

Embed.create()
  .title("Short titles are fine")
  .description("Long literal descriptions are checked when they appear in source")

Dynamic strings are not fully evaluated. If your embed description comes from variables, database content, API data, or joined arrays, validate it at runtime too.

VS Code Intelligence

The VS Code extension contributes language support for `.yuri` files and registers a Discord completion provider. The provider is source-pattern based: it checks recent text near the cursor and suggests relevant completions for common Discord shapes.

Typing ContextSuggested Items
client.on("...Event names such as `ready`, `interactionCreate`, `messageCreate`, `guildMemberAdd`, `voiceStateUpdate`, shard events, and more.
interaction.Reply helpers, narrowing helpers, `user`, `guild`, `member`, `commandName`, `options`, and `customId`.
Slash.command(...).Command builder methods such as `description`, `option`, `subcommand`, `group`, `permission`, `guildOnly`, `nsfw`, `autocomplete`, and `register`.
Embed.create().Embed builder methods such as `title`, `description`, `color`, `field`, `footer`, `author`, `thumbnail`, `image`, `timestamp`, and `validate`.
Button. / Modal. / Select. / Components.Component, modal, select, and Components v2 helper suggestions.
No Discord import yetAuto-import candidates such as `Client`, `Intents`, `Embed`, `Button`, `Slash`, `Modal`, and `Component`.

The extension also registers hover and signature help for the same completion specs. It is not a full semantic language server for every Discord.js type yet.

Type Metadata Commands

The type bridge can scan installed Discord-related packages and write JSON metadata under `.yuri-cache`. It looks for packages such as `discord.js`, `@discordjs/builders`, `discord-api-types`, `@discordjs/rest`, `@discordjs/ws`, and `@tsundere/discord`. If it cannot find installed type packages, it falls back to a small built-in graph for core Tsundere Discord types.

tsundere types inspect Client
tsundere types doctor

How the VS Code Extension Keeps Types Fresh

When the extension activates in a workspace, it schedules `tsundere types sync` unless `.yuri-cache/discord.cache.json` already exists on workspace open. It also watches `package.json`, `pnpm-lock.yaml`, `package-lock.json`, `node_modules/@tsundere/discord/package.json`, and `node_modules/discord.js/package.json`; when those files change, it schedules another sync in a terminal named `Tsundere Types`.

That means type metadata can lag behind package changes until sync runs. If completions or inspect output seem stale, run `tsundere types sync` yourself.

What Is Not Built Yet

Older roadmap language can make Discord Intelligence sound broader than the current code. These items are still future work or only partial foundations:

  • No VS Code dashboard currently summarizes all intents, permissions, OAuth scopes, and command risks.
  • `tsundere doctor` checks platform/runtime/package optimizer state, but it does not yet report Discord API compatibility or deprecated Discord APIs.
  • Role hierarchy checks are warnings and guidance only; Tsundere cannot know your server role order at compile time.
  • OAuth scope analysis is not implemented as a CLI report.
  • Deprecated API detection depends on extracted type docs and is not yet a full migration assistant.
  • Compiler diagnostics are regex/source-pattern checks, not a full TypeScript semantic analysis pass.

Troubleshooting

No Discord diagnostics appear

Check `strict` in `tsundere.config.json`. Discord diagnostics are added through the strict path. Also check `diagnostics.warnings`; warning diagnostics can be hidden if warnings are disabled.

Intent warning appears even though the bot works

The compiler only searches source text for intent names. If you build intents dynamically or import them from another file, the warning may be conservative. Keep the warning in mind, then verify the final client config and Discord developer portal settings.

`types sync` cannot find Discord packages

Run `tsundere install` in the project, or install the Discord runtime and dependencies. For a Discord project, `tsundere types doctor` will suggest `tsundere add @tsundere/discord discord.js` when no installed type sources are found.

VS Code completions look generic

The completion provider uses local text context. Add an `@tsundere/discord` import or write the expected builder shape, then try again. Restart the extension window if the provider was not activated for `.yuri` files.
