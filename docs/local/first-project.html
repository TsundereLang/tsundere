Getting Started

First Project

This page walks through the real starter flow created by `templateFiles` in `src/cli.ts`: a Discord project with `.yuri` source, config, `.env.example`, local runtime dependency, and command discovery settings.

Create the Starter

tsundere create my-bot --template discord
cd my-bot

The create command refuses to overwrite an existing folder. It writes `package.json`, `tsundere.config.json`, `.gitignore`, `.env.example`, `src/main.yuri`, and for the Discord template, `src/commands/ping.yuri`. It also tries to install the bundled Discord runtime into `.tsundere/runtime/discord` when the CLI has built runtime files available.

The folder name is resolved inside your current directory by the path safety helper. That means `tsundere create my-bot` creates a sibling folder under the shell location you are already in, while traversal outside the project root is rejected. Pick a short folder name without shell-special characters, especially on Windows where spaces and synced desktop paths can make debugging harder than it needs to be.

The Discord template currently imports `Client`, `Intents`, `Slash`, and `Embed` from `@tsundere/discord`. It creates a simple client, responds to `/ping`, and includes command discovery config. Other templates exist for REST, websocket, microservice, CLI, and empty projects, but this walkthrough uses the Discord template because it exercises the most Tsundere-specific runtime behavior.

Install Dependencies

tsundere install

This wraps npm install. Before npm runs, Tsundere checks whether the project depends on `@tsundere/discord`; if it does, the CLI repairs `.tsundere/runtime/discord` so the file dependency points at a real local package. After npm succeeds, the package optimizer stores reusable package files and writes Tsundere YAML snapshots.

You should still expect normal npm files. `package-lock.json` is npm's lockfile and Tsundere preserves it. The extra `tsundere-lock.yaml` and `tsundere-workspace.yaml` files are Tsundere's optimizer snapshot of what npm installed; they do not replace npm's lockfile. If npm fails, fix the npm error first. Tsundere only harvests packages into the store after npm exits successfully.

A successful install prints elapsed time, cache hits and misses, reused package count, stored package count, and link/copy metrics. The first install usually has misses because the store is empty. Reinstalling the same dependency set should show more reuse once the store has valid entries.

Add Your Discord Token

The starter uses `env.DISCORD_TOKEN` in `src/main.yuri`. During runtime, `src/compiler/project.ts` reads `.env`, merges values into `process.env`, and starts Node with that environment. Copy `.env.example` to `.env` and put your bot token there.

DISCORD_TOKEN=replace-me

Do not commit `.env`. The starter `.gitignore` includes `.env`, `node_modules/`, `build/`, and `.tsundere/`.

The `.env` parser is intentionally simple: it reads non-empty lines, ignores comment lines that start with `#`, splits on the first `=`, trims the key and value, and removes one surrounding pair of quotes. It does not implement every feature from dotenv packages. Keep the token on one line and avoid inline comments after the value.

Run Development Mode

tsundere dev

`tsundere dev` loads config, validates it, writes command metadata, builds the project, starts `.tsundere/runtime-build/main.js`, and watches `.yuri` files. On file changes, it kills the old runtime process, rebuilds, and starts the new output.

The generated runtime is ordinary JavaScript executed by Node. If your bot crashes, you are debugging the generated runtime and your source logic, not a hidden server. Start by reading the last error line, then inspect `.tsundere/runtime-build/main.js` if the error points at generated output. For source-level mistakes, the compiler diagnostics printed before the runtime starts are usually more useful.

Expected Output

A successful dev start usually prints a discovered command count, build messages, and your bot's runtime logs. If the token is valid and Discord gateway access works, the starter logs `Online as ...` from the `ready` event. If Discord login fails, the runtime exits and `tsundere dev` waits for more changes.

Build And Start Once

After the bot runs in development mode, test the production-style commands once. `tsundere build` compiles without starting the watch loop. `tsundere start` runs the built runtime and will build first if `.tsundere/runtime-build/main.js` is missing.

tsundere build
tsundere start

This catches a different class of issues than dev mode. For example, if your service relies on a file path relative to the source folder, it may behave differently once the runtime entry is under `.tsundere/runtime-build`. Prefer paths based on `process.cwd()` for project files and keep secrets in `.env` or deployment environment variables.

Troubleshooting

  • If `@tsundere/discord` cannot resolve, run `tsundere runtime install` and then `tsundere install`.
  • If no commands are discovered, check `tsundere.config.json` and confirm files live under `src/commands`.
  • If the bot never comes online, confirm `DISCORD_TOKEN` is present in `.env` and the bot has the required Discord intents enabled.
